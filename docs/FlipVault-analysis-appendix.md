## Adversarial Math Verdict

- **Invariant holds:** true (confidence: high)

**Proof:**

Let the conserved quantity be Q = reserve_lamports + Sum over 4 vaults of (lamports held in that vault's SOL-tranche). Consider a single flip of one vault whose SOL-tranche holds S lamports and TOKEN-tranche holds T virtual tokens. Track only REAL lamport movements (virtual tokens are never lamports, so they cannot affect Q).

STEP 1 (sell T): sol_out = r_sol - floor(k/(r_tok+T)). Real movement: reserve PDA -> this vault, of exactly sol_out lamports. Bookkeeping: reserve_lamports -= sol_out; the tranche that held T tokens now holds sol_out lamports (a SOL-tranche), contributing +sol_out to the vault-SOL sum. Delta_Q(step1) = (-sol_out) + (+sol_out) = 0.

STEP 2 (buy with S): tok_out = r_tok - floor(k/(r_sol+S)). Real movement: this vault -> reserve PDA, of exactly S lamports. Bookkeeping: reserve_lamports += S; the tranche that held S lamports now holds tok_out tokens (a TOKEN-tranche), contributing -S to the vault-SOL sum (it leaves the SOL sum entirely). Delta_Q(step2) = (+S) + (-S) = 0.

STEP 3 (swap flags): pure relabeling of which tranche is SOL vs TOKEN; no lamports move. Delta_Q = 0.

Total Delta_Q over a flip = 0, EXACTLY and for ANY value of sol_out and ANY rounding, because each movement is double-entry: the same integer that leaves the reserve enters a vault SOL-tranche (step1) and the same integer that leaves a vault SOL-tranche enters the reserve (step2). The conservation does not depend on sol_out equalling the exact curve price; it only depends on the credited tranche amount being the SAME integer that was debited from the reserve, and on S being the SAME integer moved back. Numerically verified: a single flip on a 103 SOL system left Q = 103000000000 unchanged (delta 0), and 200,000 random flips across 4 coupled vaults preserved Q exactly under both floor-quotient and ceil-quotient rounding (total0 == total1 == 108000000000). Hence Q is invariant under flips; it changes only by deposit (+amount into a vault SOL-tranche) and withdraw (-payout from a vault SOL-tranche). The curve product k = r_sol*r_tok is a SEPARATE invariant (only approximately constant under floor division) and must not be conflated with Q.

**Counterexamples / edge cases:**
- No counterexample to Q-conservation exists: it is exact for all rounding because credited-tranche == debited-reserve as the same integer. Confirmed over 200k random flips (delta 0).
- Rounding-direction (not a conservation break, a solvency leak): r_sol=50e9, r_tok=3e9, k=1.5e20, T=999999999, denom=r_tok+T=3999999999, k%denom=1500000009 (non-divisible). Exact sol_out=12499999990.625; spec's floor(k/denom)=37500000... gives sol_out=12499999991 (overpays house by 0.375 lamport, rounds payout UP). House-safe ceil(k/denom) gives sol_out=12499999990 (rounds payout DOWN). Q is conserved either way, but the bankroll bleeds sub-1 lamport per sell leg under the literal spec.
- u64 overflow: k = 50e9 * 3e9 = 1.5e20 = 68 bits > u64 max (1.844e19, 64 bits). A naive u64 multiply for k or for the dividend in k/(r_tok+T) wraps and destroys BOTH invariants. u128 required (u256 not).
- r_sol floored to 0: r_sol=1e9, r_tok=1e6, k=1e15, T=1e15 -> denom=r_tok+T=1000001000000 > k, so floor(k/denom)=0, post-sell r_sol=0. The next curve recompute r_tok=k/r_sol divides by zero (panic). Must guard with a min-reserve floor.
- Empty-tranche no-op is NOT clean under k-drift: with k held fixed but r_sol drifted by 1 (r_sol=50000000001, r_tok=3e9, k=1.5e20), T=0 yields sol_out = r_sol - floor(k/r_tok) = 1 (nonzero), i.e. a phantom 1-lamport flip on an empty token tranche. Only a no-op when k == r_sol*r_tok exactly.
- House-safe ceil can underflow: if k drifts ABOVE r_sol*r_tok (e.g. r_sol=10, r_tok=10, k=200, T=0), ceil(k/(r_tok+T)) = 20 > r_sol=10, so sol_out = 10-20 = -10. Without a clamp/checked_sub this underflows u64. Must clamp sol_out to [0, r_sol].

**Corrections to first analysis:**
- First analysis 'confirmed' item claims sol_out is strictly < r_sol for T>0 'since k/(r_tok+T) >= 1 in all practical states.' This is too weak: under a large-T drain k/(r_tok+T) can FLOOR TO 0 (worked example: k=1e15, denom>k -> quotient 0 -> r_sol becomes 0). So a single sell CAN drive the floored r_sol to exactly 0, contradicting the 'asymptotically approaches but does not reach zero' claim. The r_sol->0 guard is therefore mandatory, not merely precautionary.
- First analysis 'confirmed' item says empty-tranche math is 'naturally a no-op' because k/(r_tok+0)=r_sol. This holds ONLY when k == r_sol*r_tok exactly. After any floor-division drift (which the same writeup admits happens), T=0 yields a nonzero phantom sol_out (verified: drift of 1 produced sol_out=1). The empty-tranche path must be an EXPLICIT early-return (if T==0 skip sell, if S==0 skip buy), not relied upon to fall out of the formula.
- The first analysis's house-safe recommendation sol_out = r_sol - ceil_div(k, r_tok+T) is directionally correct but INCOMPLETE: under upward k-drift ceil(k/denom) can exceed r_sol, making sol_out negative and underflowing u64 (verified: -10). The recommendation must add a clamp sol_out = min(max(sol_out,0), r_sol) / checked_sub. Minor omission, but load-bearing for solvency.
- Spec wording is internally inconsistent on the canonical curve update: it states both r_tok = k/(r_sol) AND defines tokens_out = r_tok - k/(r_sol+dx) (implying r_tok -= tokens_out). These coincide for one trade but the persisted choice matters across trades. The spec must pin ONE form (recompute floor(k/independent)). The first analysis flags this correctly; I confirm it is a genuine spec ambiguity, not analyst error.
- No error in the core conservation proof of the first analysis: its double-entry argument and the claim that conservation is rounding-independent are CORRECT and I reproduce them. The 'high severity' overpay finding is correctly characterized as a solvency leak, NOT a conservation break.

**Mandatory rules:**
1. Store k as u128 and set k = (seed_sol as u128) * (init_r_tok as u128) at genesis. Set curve r_sol = seed_sol = the exact lamports transferred into the reserve PDA (decide explicitly that r_sol tracks SPENDABLE bankroll, excluding the rent-exempt minimum, so reserve -= sol_out never hits the rent floor).
2. Widen to u128 BEFORE every multiply/add-then-divide: compute (r_tok+T), (r_sol+S), (r_sol+dx), (r_tok+dy) in u128, divide k (u128) by them in u128, then narrow the resulting reserve quotient back to u64 with a checked cast. Never form r_sol*r_tok or the dividend in u64 (k = 1.5e20 = 68 bits overflows u64).
3. Round the REAL lamport payout DOWN (house-safe): sol_out = r_sol - ceil_div(k, r_tok+T) where ceil_div(a,b) = (a + b - 1)/b. The literal spec formula r_sol - floor(k/(r_tok+T)) rounds the payout UP and bleeds the reserve sub-1 lamport per sell. Clamp sol_out into [0, r_sol] and use checked_sub for reserve -= sol_out.
4. Round the VIRTUAL token credit DOWN: tok_out = r_tok - floor(k/(r_sol+S)) (fewer tokens credited = conservative for the house). Clamp tok_out >= 0.
5. Canonicalize the curve update: after each leg persist the dependent reserve as floor(k/independent) (one chosen form), never also persist a separately-subtracted value; otherwise k drift compounds. Add a test asserting |r_sol*r_tok - k| is within a small tolerance (NOT exact) after every flip.
6. Guard ALL divisions with checked_div + custom errors (no Rust panics): require r_tok+T > 0 (step1), r_sol+S > 0 (step2), r_tok > 0 before recompute r_sol=k/r_tok, r_sol > 0 before recompute r_tok=k/r_sol, total_shares > 0 in withdraw, tranche_amount > 0 OR total_shares==0 branch in deposit (key the 1:1 empty-mint branch off total_shares==0 consistently).
7. Enforce a hard minimum-reserve floor: reject or cap any flip whose sell leg would drive post-sell r_sol below a configured min_reserve (e.g. a fraction of seed_sol), preventing both r_sol floored to 0 (which bricks the k/r_sol recompute) and absurd post-drain prices. Verified: large T can floor r_sol to exactly 0.
8. Make empty tranches explicit early-returns: if T==0 skip the sell leg entirely (no curve mutation, sol_out=0); if S==0 skip the buy leg (tok_out=0). Do NOT rely on the formula yielding 0, because under floor-division k-drift T=0 produces a phantom nonzero sol_out.
9. Fix flip order as sell-first and guarantee exactly ONE vault flips per round (rand % 4); never two flips in one settle callback, since multiple flips on the shared curve are order-dependent and would be consensus-critical. Use checked_sub/checked_add on every real lamport move (reserve and vault PDAs) so any curve-vs-lamport drift surfaces as a clean error, preserving Q's integer exactness.

## Dimension: Bonding-curve math & the global invariant

**Mental model:** FlipVault runs ONE global constant-product curve: state is (r_sol = real lamports in a reserve PDA, r_tok = a purely virtual u64, k = r_sol*r_tok held constant). There is no SPL token and no standalone user buy/sell. The buy/sell formulas are invoked ONLY internally, as the two halves of a flip. A flip on a vault {SOL-tranche S lamports, TOKEN-tranche T tokens} runs sell-first: (1) sell the T virtual tokens into the curve, paying out sol_out = r_sol - k/(r_tok+T) lamports from reserve PDA -> vault PDA, advancing the curve to r_tok+=T, r_sol=k/r_tok; (2) buy with the S lamports of the SOL-tranche, getting tok_out = r_tok - k/(r_sol+S) virtual tokens, moving S lamports vault PDA -> reserve PDA, advancing to r_sol+=S, r_tok=k/r_sol; (3) swap the two tranches' asset flags so the tranche that held tokens now holds the sol_out lamports and the tranche that held S lamports now holds tok_out tokens. Shares (per-user, per-tranche, in position PDAs) are never touched by a flip; holders keep their fractional claim on whatever the tranche now contains. The global invariant the spec wants is: reserve_lamports + Sum(SOL-tranche lamports across all 4 vaults) is constant, moved only by deposit(+)/withdraw(-), never by a flip. I verified symbolically and with integers that a flip conserves this sum because every lamport movement is double-entry (reserve -= sol_out / vault += sol_out, then reserve += S / vault -= S; net sum delta = 0) and this is INDEPENDENT of any rounding. I also verified r_sol (curve field) stays equal to actual reserve_lamports after each step PROVIDED genesis sets r_sol = seed_sol exactly and the field is always recomputed as k/r-other (floor). The real risks are not in conservation (that is robust) but in (a) rounding DIRECTION of the real lamport payout sol_out, which the spec's formula rounds UP against the house, (b) u64 overflow of k which mandates u128, (c) reserve drain toward zero on sustained net-sells, and (d) division-by-zero / empty-tranche guards.

**Confirmed:**
- Constant-product formulas are correct: buy tokens_out = r_tok - k/(r_sol+dx) then r_sol+=dx, r_tok=k/r_sol; sell sol_out = r_sol - k/(r_tok+dy) then r_tok+=dy, r_sol=k/r_tok. Verified numerically: a clean round-trip (buy 5 SOL then sell the tokens back) returns exactly 5 SOL with these test integers, and tokens_out=r_tok-k/(r_sol+dx) is identical to recompute k/(r_sol+dx).
- k = r_sol*r_tok is intended to stay constant; it is the conserved curve product, NOT the same thing as the conserved lamport sum. They are different invariants.
- Buy/sell are used ONLY inside flip. There is no standalone user buy/sell instruction in the instruction list (initialize, deposit, close_round, withdraw). Users interact with the curve only indirectly when a flip executes on a vault they hold a tranche in. This reading is correct.
- Flip lamport conservation PROVEN: net reserve delta = -sol_out + S, net vault delta = +sol_out - S, sum delta = 0 exactly, for ANY rounding. reserve_lamports + Sum(SOL-tranche lamports) is conserved by every flip. Verified step-by-step with integers (sum 103000000000 before and after).
- The curve field r_sol stays equal to actual reserve PDA lamports after each flip step, because step1 moves exactly sol_out and sets r_sol-=sol_out, step2 moves exactly S and sets r_sol+=S. Verified r_sol==reserve_lamports at every checkpoint in the numeric trace.
- Empty-tranche math is naturally a no-op: T=0 gives k/(r_tok+0)=r_sol so sol_out=0; S=0 gives k/(r_sol+0)=r_tok so tok_out=0. The curve is untouched when a tranche is empty.
- sol_out = r_sol - k/(r_tok+T) is strictly less than r_sol for T>0 (since k/(r_tok+T) >= 1 in all practical states), so a single sell can never by itself set r_sol to exactly 0; it asymptotically approaches but does not reach zero.
- Withdraw payout = shares*tranche_sol/total_shares and deposit shares = amount*total_shares/tranche_amount are pro-rata; with floor division a user cannot extract more than their fair fraction.

**Findings:**
- [high/high] **Sell payout sol_out is rounded UP against the house (should round DOWN)** — sol_out = r_sol - k/(r_tok+T) using integer floor on k/(r_tok+T) makes the SUBTRAHEND smaller, so sol_out comes out >= the exact real value. The reserve therefore pays slightly MORE lamports than the exact curve price on every sell (the lamport-paying half of every flip). Numeric example: exact 16666666666.67, spec floored payout 16666666667 (overpays by up to ~1 lamport). The house-safe form is sol_out = r_sol - ceil(k/(r_tok+T)) = r_sol - (k + (r_tok+T) - 1)/(r_tok+T), which yields 16666666666 (rounds the payout DOWN). Each flip bleeds the reserve by sub-1-lamport; over many rounds and 4 vaults this is a systematic, one-directional drain on the bankroll. Conservation of the lamport SUM is unaffected (the same value moves and is recorded), but solvency margin erodes.

- [critical/high] **k = r_sol*r_tok overflows u64; u128 widening is mandatory at every multiply/divide** — With r_sol ~ 1e12 lamports (1000 SOL) and r_tok ~ 1e9, k ~ 1e21 = 70 bits, exceeding u64 (64 bits, max ~1.8e19). For larger reserves (1e16 lamports, 1e12 tokens) k ~ 1e28 = 94 bits. Therefore: k MUST be stored as u128; init must compute seed_sol as u128 * init_r_tok as u128; every k/(r_sol+dx) and k/(r_tok+dy) must be done in u128 and only narrowed back to u64 after the division (the quotient r_tok/r_sol fits u64, but the dividend k does not). u256 is NOT required. Additionally (r_sol+dx) and (r_tok+dy) are u64 sums that can overflow near u64 max, so widen the addends to u128 before the add/mul/div. A naive u64 implementation silently wraps and destroys both invariants.

- [high/high] **Reserve r_sol can be drained arbitrarily close to zero by a large TOKEN tranche; explicit floor guard required** — Because sol_out = r_sol - k/(r_tok+T), a single very large token tranche T pushes the post-sell reserve r_sol down to k/(r_tok+T). With r_tok+T = 1e12 the reserve collapses from 1e11 to 1e8 (a 1000x drain) in one flip. r_sol reaches exactly 0 only if T > r_tok*(r_sol-1) ~ k, which is practically unreachable but must still be guarded because (a) a tiny r_sol makes subsequent buy prices explode and withdraw economics absurd, and (b) if r_sol ever floored to 0, the next sell divides fine but the curve is economically dead and r_tok=k/r_sol divides by zero. The spec explicitly calls for a guard against r_sol -> 0. Recommend a hard floor (reject or cap a flip/sell that would drive r_sol below a configured minimum, e.g. a fraction of seed_sol).

- [high/high] **Division-by-zero surfaces that must be guarded** — Step1 divides by (r_tok+T): zero iff r_tok==0 AND T==0. Step2 divides by (r_sol+S): zero iff r_sol==0 AND S==0. The curve recompute r_sol=k/r_tok divides by r_tok (zero if r_tok floored to 0) and r_tok=k/r_sol divides by r_sol (zero if r_sol floored to 0). Withdraw divides by total_shares (zero if the tranche has no shares). Deposit pro-rata divides by tranche_amount (the spec covers this with the 1:1 empty branch, but the branch condition must key off total_shares==0 OR tranche_amount==0 consistently). Each of these needs an explicit checked/guarded path; an unchecked div in Rust panics and bricks the instruction.

- [medium/high] **Curve consistency depends on ALWAYS storing r as floor(k/other); storing r -= out instead causes compounding drift** — The spec ambiguously states both forms for the post-trade reserve (e.g. buy: r_tok = k/r_sol, but tokens_out was computed as r_tok - k/(r_sol+dx), implying r_tok -= tokens_out). I verified these two are identical at the moment of one trade. The danger is across trades: if the implementation stores r_tok -= tokens_out (subtract style) rather than r_tok = k/r_sol (recompute style), then because the OTHER reserve was floored on the previous trade, r_sol*r_tok no longer equals k and the discrepancy compounds. Over 100k random trades I measured k dipping below k0 (min k 99999999832648989600 vs k0 1e20) under recompute already; subtract-style would drift further and unpredictably. Pick ONE canonical update (recommend: always recompute the dependent reserve as floor(k/independent) and never persist a separately-derived value) and assert r_sol*r_tok within tolerance of k.

- [high/medium] **Genesis must set curve r_sol exactly equal to seed_sol, or the invariant is broken from block zero** — The lamport invariant and the no-underflow safety of step1 (reserve -= sol_out, which is safe only because sol_out <= r_sol == reserve_lamports) both require the curve field r_sol to equal the actual lamports in the reserve PDA. initialize(seed_sol, init_r_tok, ...) must therefore set r_sol = seed_sol and k = seed_sol * init_r_tok (u128). If the design ever lets the founder pass an independent r_sol distinct from the deposited seed_sol, or if rent-exempt lamports in the reserve PDA are counted as part of r_sol, the field drifts above real spendable lamports and the very first large flip will underflow reserve -= sol_out (panic) or, worse, over-credit a vault. Decide explicitly whether r_sol tracks total PDA lamports including rent or only the spendable bankroll above rent-exemption.

- [info/medium] **Within-flip round-trip is NOT value-neutral and that is intended, but the direction matters for fairness** — A flip sells T then buys with S on the same curve, so the two operations move the price against each other. This is the core game mechanic (a flipped vault's holders are repriced), not a bug. However note the sequencing asymmetry: selling T first DROPS r_sol, making the subsequent buy-with-S cheaper (more tok_out). A buy-first ordering would yield different tranche contents. The spec fixes sell-first; implementers must not reorder. Also, because both legs touch the global curve, the four vaults are coupled: the order vaults are flipped across rounds changes outcomes. Confirm this global coupling is intended (it is implied by 'one shared curve') and that only ONE vault flips per round (rand % 4) so there is no intra-round ordering ambiguity.

- [low/high] **Two distinct invariants must not be conflated in tests** — Test code must assert TWO separate things: (1) the LAMPORT invariant reserve_lamports + Sum(SOL-tranche lamports) changes only by deposits/withdrawals (exact integer equality, robust to all rounding), and (2) the CURVE invariant r_sol*r_tok stays within a small tolerance of k (NOT exact, because floor division perturbs it). Asserting exact k equality will fail; asserting only the curve invariant will miss lamport leaks. The spec names only the lamport invariant; add the curve-tolerance assertion explicitly.

**Open questions:**
- Does the curve field r_sol track the TOTAL lamports in the reserve PDA (including the rent-exempt minimum) or only the spendable bankroll above rent-exemption? This determines whether step1's reserve -= sol_out can hit the rent floor and fail, and whether r_sol==reserve_lamports is literally true or off by the rent reserve.
- When the spec writes both 'r_tok = k/(r_sol)' and implies 'r_tok -= tokens_out', which is the canonical persisted update? Recommend recompute-from-k, but this must be pinned down to avoid compounding curve drift.
- What is the intended rounding direction for sol_out? The literal formula rounds the real payout UP (against the house). Was that intentional (favoring token holders) or should it be ceil-of-the-quotient (round payout DOWN, house-safe)? This is a value-flow decision, not just a precision detail.
- What concrete guard is wanted for r_sol -> 0? A hard minimum reserve floor, a per-flip cap on T, or rejection of flips that would drop r_sol below X% of seed_sol? The spec says 'guard against r_sol -> 0' without specifying the policy.
- Is it guaranteed that exactly one vault flips per round (rand % 4), and never two in the same round? If multiple flips could occur in one settle callback, their ordering on the shared curve becomes consensus-critical and must be deterministic.
- For the buy step, tok_out = r_tok - k/(r_sol+S) is purely virtual (no lamports), but it determines the new TOKEN-tranche amount that future withdrawals/flips depend on. Should tok_out round DOWN (fewer virtual tokens credited, conservative) and is that the intended direction for the token-tranche holders?
- What units/scale are init_r_tok and seed_sol expected in, and what bounds does initialize enforce so that k stays within u128 and r_sol stays large enough that early flips do not round small payouts to zero?

**Recommendations:**
- Store k as u128 and perform EVERY multiply, add-then-divide in u128: widen r_sol, r_tok, dx/dy, S, T to u128, compute k/(r_tok+T) and k/(r_sol+S) in u128, then narrow the resulting reserve back to u64 with a checked cast. Never compute r_sol*r_tok in u64.
- Round the REAL lamport payout DOWN: implement sell as sol_out = r_sol - ceil_div(k, r_tok+T) where ceil_div(a,b) = (a + b - 1)/b. This guarantees the house is never overpaid and removes the systematic sub-lamport reserve bleed. Keep virtual tok_out floored (round token credit down) so the token side is also conservative for the house.
- Canonicalize the curve update: after each leg, set the dependent reserve to floor(k/independent) and persist ONLY that; do not also persist a separately-subtracted value. Add a debug_assert / test that r_sol*r_tok is within a tiny epsilon (e.g. < r_sol + r_tok) of k after every flip.
- In initialize, set r_sol = seed_sol (the lamports actually transferred into the reserve PDA) and k = (seed_sol as u128) * (init_r_tok as u128). Validate init_r_tok > 0 and seed_sol > 0, and bound them so k fits u128 with margin. Document whether r_sol includes rent.
- Add an explicit reserve floor: reject or cap any flip whose sell leg would drive r_sol below a configured minimum (e.g. a stored min_reserve = some fraction of seed_sol). Combined with checked_sub on reserve -= sol_out, this prevents both economic death and arithmetic underflow.
- Guard every division: require r_tok+T > 0, r_sol+S > 0, total_shares > 0 (withdraw), and use checked_div everywhere returning a custom error instead of panicking. Make deposit's empty branch trigger on total_shares==0 (1:1 mint) consistently.
- Write the test suite to assert BOTH invariants separately: (1) exact integer equality of reserve_lamports + Sum(SOL-tranche lamports) across all flips (allowing change only on deposit/withdraw), and (2) curve-product r_sol*r_tok within tolerance of k. Include adversarial cases: T=0, S=0, T enormous (drain test), repeated flips on the same vault, and a deposit-then-immediate-flip in the same round.
- Use checked_sub for reserve -= sol_out and vault -= S, and checked_add for the credit sides, so any drift between the curve field and real lamports surfaces as a clean error rather than silent corruption.

## Dimension: Anchor / Solana Account & PDA Architecture

**Mental model:** FlipVault is a single-program system with a small fixed set of singleton accounts plus an unbounded set of tiny per-user Position accounts. There are exactly four "fund" accounts that hold real lamports: one Reserve PDA (the house bankroll = r_sol) and four Vault PDAs (each holds the lamports of whichever of its two tranches is currently SOL-denominated). All of these are program-owned, data-carrying accounts, so lamports cannot be moved with system_program::transfer (the System Program can only debit accounts it owns, i.e. accounts with no data and owner == System Program). Every reserve<->vault movement, and the vault side of withdraw/deposit payouts, must be done by directly mutating the two AccountInfo lamport fields under try_borrow_mut_lamports, conserving the sum. Only user->program inflows (deposit, initialize seed) and program->user outflows (withdraw) cross the program boundary; deposit-in can use a CPI system transfer from the user wallet (which IS system-owned) into the Vault, but the withdraw payout must be a manual lamport debit from the Vault.

The state model splits cleanly:
- Config/Curve (global singleton): r_tok (virtual u128), k (u128), r_sol is NOT stored as the source of truth here — the true r_sol is the Reserve PDA's actual lamports minus its rent floor; storing a redundant r_sol invites drift, so Config should store only r_tok, k, round_secs, last_round_ts, a VRF/pending-settle state machine, the program authority/founder, and bumps.
- Reserve PDA: holds r_sol as real lamports. Either a bare SystemAccount-style PDA owned by the program with no Anchor data (cleanest for the "lamports == r_sol" identity) or a thin data account; I recommend a data account with a bump and discriminator so it can be validated by seeds, accepting that its rent-exempt minimum is a fixed floor below which r_sol must never fall.
- 4 Vault PDAs: each stores both tranches' {asset_flag, amount, total_shares} and physically holds the SOL-tranche's lamports. amount for the SOL tranche is the "logical" SOL accounting; the vault's real lamports == that SOL tranche amount + rent floor.
- Position PDAs: one per (user, vault, tranche_index in {0,1}), storing owner pubkey, vault, tranche index, and shares (u64). Non-transferable because the program never exposes any instruction that reassigns owner, and every mutating instruction asserts position.owner == signer.

A flip never crosses the program boundary: it sells the TOKEN tranche back to the curve (reserve -> vault lamports) and buys with the SOL tranche (vault -> reserve lamports), then swaps the two tranches' asset flags while leaving total_shares and all Position.shares untouched. That is why the global invariant (reserve_lamports + sum of SOL-tranche lamports) is conserved across flips and only moves on deposit/withdraw.

**Confirmed:**
- There is exactly one global constant-product curve (r_sol, r_tok, k) shared across all 4 vaults; r_sol is real lamports in a Reserve PDA, r_tok and k are virtual numbers.
- There are exactly 4 vaults, each with 2 tranches; at any instant one tranche is SOL-denominated and the other TOKEN-denominated (either may be empty). Each tranche stores {asset, amount, total_shares}.
- Per-user balances are non-transferable Position PDAs keyed by (user, vault, tranche) holding a share count.
- Only SOL moves as real lamports; the game token is purely a u64 inside the curve with no SPL mint.
- Flip is sequential sell-first: sell TOKEN tranche T to the curve (reserve->vault), then buy with SOL tranche S (vault->reserve), then swap the asset flags; shares are untouched so holders keep their fractions.
- The conservation invariant is reserve_lamports + sum(SOL-tranche lamports across 4 vaults) == constant, changed only by deposits (+) and withdrawals (-), never by flips.
- close_round runs at most once per round_secs, requests Switchboard On-Demand VRF, and in the settle callback picks vault = rand % 4 and flips it.
- withdraw is only permitted from a SOL tranche; TOKEN-tranche positions must wait for the vault to be re-selected and flipped back to SOL.
- deposit adds SOL to the vault's current SOL tranche and mints shares pro-rata (amount * total_shares / tranche_amount, or 1:1 if empty); lamports go to the Vault PDA and the deposit is immediately eligible for the current round's flip.
- initialize funds the Reserve PDA with seed_sol and sets r_tok and k; the reserve is the house bankroll and must be guarded against r_sol -> 0 on sustained net-sells.

**Findings:**
- [critical/high] **Reserve and Vault PDAs carry data, so system_program::transfer cannot debit them — must use direct lamport mutation** — This is the load-bearing Solana mechanic the spec correctly flags. The System Program will only debit an account whose owner is the System Program and which has zero data; once the program assigns/owns Reserve and Vault and writes a discriminator + bump, a CPI transfer that tries to debit them fails with 'instruction spent from the balance of an account it does not own'. Correct pattern for reserve<->vault (flip) and for the withdraw payout: directly mutate AccountInfo lamports. Concretely:

  let mut from = from_ai.try_borrow_mut_lamports()?;
  let mut to   = to_ai.try_borrow_mut_lamports()?;
  **from = from.checked_sub(amount).ok_or(Underflow)?;   // also assert *from - amount >= rent_floor
  **to   = to.checked_add(amount).ok_or(Overflow)?;

Both accounts must be mutable, program-owned, and the debit must be checked against the rent-exempt floor so the account is never left below rent (which would make it purgeable and could destroy state). The deposit INFLOW is the asymmetric case: the user wallet is System-owned, so deposit-in should be a CPI system_program::transfer from the signer into the Vault (allowed, since you are crediting a program account, not debiting it). Only the withdraw payout and all reserve<->vault shuttling use the manual debit pattern.

- [high/high] **Storing r_sol in Config duplicates the Reserve's lamport balance and will drift from the manual-mutation path** — The spec's curve math debits/credits r_sol on every buy/sell/flip. If r_sol lives both as a Config field AND as the Reserve PDA's real lamports, the two can diverge: e.g. a rent top-up, a stray airdrop to the Reserve PDA, or a mismatch between the computed sol_out and the actual lamport delta silently breaks the invariant. Recommendation: treat the Reserve PDA's (lamports - rent_floor) as the single source of truth for r_sol, recompute k-consistency from r_tok, and never store a second r_sol. If a cached r_sol is kept for compute efficiency, add a debug-only assert_eq!(cached_r_sol, reserve.lamports - rent_floor) in tests, and recompute the actual lamport move from the curve formula rather than trusting the cache.

- [high/high] **close_round / settle mutates global Config but the flip touches a specific Vault — the settled vault is not known at request time** — VRF is two-phase: close_round requests randomness; the settle callback computes vault = rand % 4 and flips it. At request time you do not yet know which of the 4 vaults will be selected, so you cannot pass a single Vault account into the request. Two viable account-architecture options: (A) the settle/callback instruction takes ALL 4 Vault PDAs as mutable accounts (plus Reserve and Config) and indexes into them by rand % 4 — simplest, deterministic account set, but pays for 4 writable account locks; (B) settle is permissionless and the caller passes the one Vault implied by the revealed randomness, with the program re-deriving rand % 4 from the stored VRF result and asserting the passed vault's seed index matches — fewer locks but requires the program to verify the supplied vault is the correct one (seeds = [VAULT, &[selected_index]] with selected_index recomputed on-chain). I recommend (A) for safety/auditability unless compute/lock pressure forces (B). Either way Config must hold a pending-settle flag + the VRF account/result so a second close_round in the same round is rejected and settle is idempotent (cannot flip twice off one draw).

- [high/high] **Rent-exempt floor must be reserved on Reserve and every Vault, and the curve's r_sol must be measured ABOVE that floor** — A data account below its rent-exempt minimum becomes eligible for garbage collection. Reserve and Vaults always carry a discriminator + fixed fields, so each has a fixed rent_floor = Rent::minimum_balance(account_data_len). The accounting must be: logical r_sol = reserve.lamports - reserve_rent_floor; logical vault SOL-tranche amount tracks (vault.lamports - vault_rent_floor). Every manual debit must assert post-balance >= rent_floor. The spec's own 'guard against r_sol -> 0' becomes 'guard against reserve.lamports dropping to rent_floor' — sustained net-sells move lamports reserve->vault and can drain the reserve toward its floor, at which point further sells (flips selling TOKEN back) must be bounded so sol_out never exceeds reserve.lamports - rent_floor. This bound is a hard on-chain check, not just a test invariant.

- [high/medium] **Position PDA non-transferability is enforced by absence of a transfer instruction plus owner == signer on every mutator, but init_if_needed introduces a re-init / share-confusion risk** — Non-transferability is structural: there is no instruction that rewrites Position.owner, and deposit/withdraw both constrain has_one = owner (or seeds = [POSITION, owner.key(), vault.key(), &[tranche_index]] with the signer == owner). That is correct. The subtlety is deposit needing init_if_needed for a user's first deposit into a tranche while reusing the PDA on later deposits: init_if_needed must (a) be feature-gated and carefully audited because it does NOT re-run init logic if the account already exists, so you must not assume fields are zeroed; (b) guard against the asset-flag mismatch — a Position is bound to a tranche INDEX (0/1), but which asset that index holds flips over time. Because the spec keys positions by (user, vault, tranche) and flips swap only the asset flags (not the share ledgers), a position index must stay glued to its share ledger across flips; deposit into 'the current SOL tranche' means resolving which index currently has asset==SOL and writing the position for THAT index. Mis-resolving the index on init_if_needed would credit shares to the wrong ledger.

- [medium/high] **Position keyed by (user, vault, tranche-index) vs (user, vault, asset) is a real design fork the spec does not pin down** — The spec says positions are keyed by (user, vault, tranche). A tranche is a fixed slot (index 0 or 1) whose ASSET flag flips each time the vault is selected. If positions are keyed by tranche INDEX, then after a flip a holder who deposited SOL now owns shares of the now-TOKEN tranche (correct per 'holders keep their fractions'), and the index is stable — good. If instead someone keys by ASSET (SOL-position vs TOKEN-position), the key would have to migrate on every flip, which is impossible for a PDA seed. So tranche must mean a stable index, and the PDA seed must use the index byte, not the asset. This needs to be stated explicitly before coding because getting it backwards breaks withdraw eligibility checks.

- [low/medium] **Deposit 'immediately eligible — if the vault flips this round, it flips' implies no per-round snapshot/epoch gating on shares** — Because shares are pro-rata of tranche_amount and flips leave shares untouched, a deposit made microseconds before settle participates fully in the flip. This is fine mathematically but means there is no need for an epoch/cursor field on positions or tranches for correctness — however it also means a MEV/timing surface: a depositor can observe the VRF request is pending and deposit into the about-to-be-selected vault. Account architecture cannot fix this (it is a game-design choice), but note that NO per-round accounting account is required, which simplifies the model: tranche state needs only {asset, amount, total_shares}, no round cursor.

- [low/medium] **Four writable Vault locks + Reserve + Config + VRF account in settle approaches the transaction account/lock budget** — If settle takes all 4 Vaults (writable), Reserve (writable), Config (writable), the Switchboard VRF/randomness account, plus system_program and the settle caller, that is ~9 accounts, all but VRF/system writable. This fits within limits but the 4 writable vault locks serialize against any concurrent deposit/withdraw on those vaults during the settle slot. Acceptable, but worth confirming compute budget for 4 deserialize + the curve math (u128 mul/div) stays under the CU cap; the alternative (B) single-vault settle reduces locks at the cost of on-chain vault-index verification.

**Open questions:**
- Is a 'tranche' a stable index (0/1) whose asset flag flips, or an asset identity? PDA seeds force it to be a stable index — confirm and document, because the position-eligibility logic depends on it.
- Should the Reserve PDA be a bare lamport-only program account (cleanest 'lamports == r_sol' identity) or a discriminated data account with a bump? This determines the rent floor and how r_sol is read.
- Is r_sol stored redundantly in Config, or derived from reserve.lamports - rent_floor? Recommend derived; needs a decision before the curve math is wired.
- Does settle pass all 4 Vaults (option A) or a single program-verified Vault (option B)? Affects the account context for the VRF callback and the lock footprint.
- What exactly is the on-chain lower bound for r_sol on a sell/flip — reserve.lamports - rent_floor, or a configured min_reserve buffer above the rent floor to keep prices sane? The spec says 'guard against r_sol -> 0' but not the numeric floor.
- Position creation uses init_if_needed on deposit — is that acceptable given its re-init footguns, and who pays the position rent (the depositing user, presumably)?
- Is withdraw allowed to fully drain (and thus close) a tranche/position, reclaiming position rent to the user, or must a dust/rent floor remain? Affects whether withdraw can close the Position account.
- Does k stay strictly constant, or is it recomputed (k = r_sol * r_tok) after rounding so integer truncation does not let k drift downward over many flips? This is a state-field-precision decision (store k as u128, recompute vs persist).
- How is the Config authority/founder gating defined for initialize and for permissioned vs permissionless close_round/settle?

**Recommendations:**
- Account inventory and seeds: Config PDA seeds=[b"config"] (singleton); Reserve PDA seeds=[b"reserve"]; Vault PDAs seeds=[b"vault", &[vault_id]] for vault_id in 0..4; Position PDAs seeds=[b"position", owner.key().as_ref(), &[vault_id], &[tranche_index]]. Store each PDA's bump in its own state (canonical bump) and validate with bump = <stored> in the seeds constraint to avoid recomputation cost.
- Config state (store only the truth, ~ size): 8 disc + 32 authority + 16 r_tok(u128) + 16 k(u128) + 8 round_secs(i64/u64) + 8 last_round_ts(i64) + 32 pending VRF account + 1 settle_phase enum + 1 selected_vault(Option u8 -> 2 bytes) + 1 config_bump + 1 reserve_bump + padding => round to ~128 bytes. Do NOT store r_sol here; derive it from the Reserve PDA.
- Reserve PDA: prefer a discriminated data account holding only {bump} (8 + 1, pad to ~16 bytes) so it is seed-validatable; its rent floor is Rent::minimum_balance(16). Treat logical r_sol = reserve.lamports - that floor. If you instead make it lamport-only/System-owned you cannot manually debit it from the program — so it MUST be program-owned to support try_borrow_mut_lamports.
- Vault PDA state: 8 disc + per-tranche {asset:u8, amount:u64, total_shares:u64} x2 = 34 + bump 1 + vault_id 1 => ~44 bytes, pad to 64. Its rent floor = Rent::minimum_balance(64). Enforce logical SOL-tranche amount == vault.lamports - rent_floor as a test invariant and recompute lamport moves from the curve, not from stored amount alone.
- Position PDA state: 8 disc + 32 owner + 1 vault_id + 1 tranche_index + 8 shares + 1 bump => ~51 bytes, pad to 64; rent floor = Rent::minimum_balance(64). The depositing user pays this rent on first deposit (init or init_if_needed with payer = signer).
- Lamport movement: implement a single helper fn move_lamports(from: &AccountInfo, to: &AccountInfo, amount: u64, from_rent_floor: u64) that asserts from.lamports() - amount >= from_rent_floor, then mutates via try_borrow_mut_lamports with checked_sub/checked_add. Use it for the two flip transfers (reserve<->vault) and for the withdraw payout (vault->user, where user is System-owned and CAN be credited directly). For deposit INFLOW, use CPI system_program::transfer from signer -> vault (do NOT manually mutate, because the source is the user's System-owned wallet and needs a signed transfer).
- Curve precision: store r_tok and k as u128; on every buy/sell/flip recompute the integer result with mul/div in u128, then optionally re-derive k = r_sol * r_tok after the move and persist it so truncation cannot let k drift down across thousands of flips. Compare computed sol_out against (reserve.lamports - rent_floor) and clamp/abort to honor the r_sol floor guard.
- Settle/VRF account context: choose option A — settle takes Config(mut), Reserve(mut), all four Vault PDAs(mut), the Switchboard randomness account, system_program. Store a settle_phase state machine in Config (Idle -> Requested(vrf_pubkey, slot) -> done) so a duplicate close_round in the same round_secs window is rejected and settle is idempotent (clears phase to Idle and stamps last_round_ts only after a successful single flip).
- Non-transferability: never expose any owner-reassigning instruction; on deposit and withdraw use has_one = owner (or signer == position.owner) plus the seed binding owner.key() so the PDA address itself proves ownership. This makes positions soulbound by construction.
- Anchor constraints to wire: Config has_one = authority for privileged ops; Vault validated by seeds=[b\"vault\", &[vault_id]] with the vault_id also stored in state for cross-check; Position uses init_if_needed(payer = signer, space = 64, seeds = [...]) on deposit and a plain mut Account on withdraw; on a full withdraw that zeroes shares, optionally close = owner the Position to reclaim rent. Add an on-chain check that the resolved 'current SOL tranche index' matches the tranche_index in the Position seed before crediting/debiting shares, so deposits and withdraws can never hit the wrong tranche ledger after a flip.
- Realloc: none of these accounts grow, so no reallocation is needed — all are fixed-size; explicitly size them up-front to avoid init_if_needed space mismatches.

## Dimension: Switchboard On-Demand VRF Integration (current 2026 API)

**Mental model:** FlipVault needs one unbiased number per round to pick which of 4 vaults flips (vault = rand % 4). The spec describes this as "close_round requests VRF, then a settle callback picks the vault and flips." That mental model is inherited from the DEPRECATED switchboard-v2 VRF (a true async oracle callback that CPIs back into your program). The CURRENT mechanism — Switchboard On-Demand randomness (crate `switchboard-on-demand`, account type `RandomnessAccountData`) — is NOT a callback. It is a 2-transaction commit-then-reveal protocol driven by the client, with NO program callback:

1. CREATE: A `Randomness` account is created once (client-side `sb.Randomness.create(program, kp, queue)`), owned by the Switchboard On-Demand program and bound to a Switchboard `queue`. It can be reused across many rounds.

2. COMMIT (tx #1): The client builds `randomness.commitIx(queue)` and runs it in the SAME transaction as your own program instruction (your "close_round"). `commitIx` binds the randomness account to the CURRENT recent slothash (a future-to-the-oracle seed nobody can predict yet). Your close_round instruction here just records "a round is now pending, committed at slot/time X" and asserts the randomness has NOT already been revealed (defensive). No random value exists yet.

3. The Switchboard oracle (running in a TEE) observes the committed slot once it is finalized and generates the random value seeded from that slothash. Neither the user nor the oracle could predict it at commit time.

4. REVEAL / SETTLE (tx #2, a different transaction, ~1+ slot later): The client builds `randomness.revealIx()` and runs it in the SAME transaction as your "settle" instruction. Inside settle, your program parses the account with `RandomnessAccountData::parse(account.data.borrow())` and calls `get_value(clock.slot)`, which returns a 32-byte `[u8; 32]` array on success, or an Err if the value is not yet resolved. You then compute `vault = revealed[0] % 4` (or fold more bytes for better uniformity) and run the flip.

So the spec's "two-step close_round (request VRF then settle callback)" maps cleanly onto commit (tx1) + reveal/settle (tx2), with the crucial correction that step 2 is a SECOND client-submitted transaction, NOT an oracle-initiated CPI callback. The program never gets called by the oracle; the program is only ever called by whoever submits the reveal tx, and it reads an account the oracle wrote to. The vault selection and flip MUST live in the settle/reveal instruction, not in close_round/commit. The commit-reveal split is also what makes the game fair: collateral/positions are locked at commit (deposit eligibility frozen for the round), and the outcome is only knowable at reveal — preventing the committer from peeking and aborting an unfavorable result.

**Confirmed:**
- The game needs exactly one fresh random u64/byte per round to compute vault = rand % 4 and flip that vault. No randomness is consumed anywhere else.
- Only SOL moves as real lamports; the curve token is virtual. VRF only selects WHICH vault flips - it does not touch curve math or lamport amounts, so a biased/replayed value affects only fairness of selection, not solvency arithmetic.
- The spec frames close_round as request-VRF-then-settle-callback, explicitly acknowledging two steps and a settle phase where vault = rand % 4 is chosen and the flip executed.
- The round cadence is gated: close_round is allowed once per round_secs (=30s), which must be enforced against Clock to prevent spamming VRF requests / re-rolling.
- The flip itself (sell-first, then buy, swap asset flags) is deterministic given the selected vault; randomness only chooses the index 0..3.

**Findings:**
- [critical/high] **Spec assumes a true async VRF callback; On-Demand has no program callback (architecture must be 2-tx commit/reveal)** — The spec says 'request VRF, then in the settle callback pick vault and flip.' Current Switchboard On-Demand (crate `switchboard-on-demand`, type `RandomnessAccountData`) is commit-then-reveal across TWO separate client transactions; the oracle writes the value into the Randomness account but NEVER CPIs back into FlipVault. There is no callback handler. Implication: split close_round into TWO instructions: (a) `close_round_commit` run in the same tx as `randomness.commitIx(queue)` - it marks the round pending and stores commit slot/time; (b) `settle_round` run in the same tx as `randomness.revealIx()` - it parses the account, reads `get_value(clock.slot)`, computes `vault = value[0] % 4`, and performs the flip. Building this as a v2-style callback will not compile against the current API and will waste the integration.

- [high/high] **Strict freshness check (seed_slot == clock.slot-1) is incompatible with a 30s round unless reveal is prompt** — The canonical Anchor example asserts `randomness_data.seed_slot != clock.slot - 1 => RandomnessExpired`, i.e. reveal must land in the slot immediately AFTER the committed slot (~400-800ms later). The oracle-side hard expiry is ~1 hour. These are two different windows: the tight per-slot check is app-imposed for maximum freshness; the 1-hour is the protocol max. For FlipVault's 30s rounds you must DECIDE the reveal window. The strict `clock.slot - 1` rule is brittle (if the reveal tx misses that exact next slot, the round bricks and must be re-committed). Recommendation: do NOT hard-bind to `clock.slot - 1`. Instead rely on `get_value(clock.slot)` succeeding (oracle revealed) plus your own round-level guard that the reveal happens within round_secs of commit. Be explicit that randomness expires (~1h) if never revealed.

- [high/high] **Grinding / selective-abort risk: whoever submits the reveal tx sees the outcome before it lands and can choose not to submit it** — Because reveal is a normal client tx (not a forced callback), the submitter can simulate `settle_round` off-chain, see which vault would be selected (and whether the flip helps or hurts their own token-tranche position), and simply NOT broadcast the reveal if unfavorable - leaving the round stuck until someone else (or a timeout path) settles. This is the classic selective-reveal attack. Mitigations: (1) make settle PERMISSIONLESS so any keeper can force it, removing the abort advantage; (2) freeze deposit eligibility and any position changes at commit so the outcome cannot be front-run by the committer adjusting exposure between commit and reveal; (3) since the SAME randomness account is committed to a fixed slothash, an aborted reveal cannot be 're-rolled' for a better number without a new commit at a new slot - so add a guard that a pending round can only be re-committed after it has provably expired, never simply re-rolled.

- [high/high] **Re-commit/timeout path is unspecified - a non-revealing oracle bricks the round** — The spec has no handling for 'oracle never reveals.' If the committed slothash is never revealed within the ~1h expiry, `get_value` will error forever and the round cannot settle. Deposits made for that round (which the spec says are immediately eligible and lock in) would be stuck if you block all activity on a pending round. Need an explicit `cancel_round` / `recommit_round` instruction: after a timeout (>= some multiple of round_secs, and within or past the 1h expiry), allow re-committing the same Randomness account to a fresh slothash (new commitIx) for the same logical round, OR cancel the round (no flip) and resume. Without this, a single oracle hiccup permanently freezes the protocol.

- [low/high] **vault = rand % 4 using only one byte is fine for 4 outcomes but document the byte selection** — `get_value` returns `[u8; 32]`. Using `value[0] % 4` over 4 vaults: 256 % 4 == 0, so the distribution is exactly uniform (each vault gets 64 of 256 byte values) - no modulo bias for n=4. This is acceptable. However, be explicit and deterministic about WHICH byte(s) you fold (e.g. always `value[0]`), and never derive selection from anything the committer controls (no mixing in user pubkeys/amounts), or you reintroduce grinding. For future-proofing (if vault count changes to a non-power-of-2), fold all 32 bytes into a u64/u128 and reduce, or use rejection sampling.

- [high/high] **Clock-based once-per-round guard must use last_settled_time, not last_commit_time, and bind randomness account identity** — To enforce once-per-round_secs you must store the round's timing on a global/curve PDA. Two subtle bugs to avoid: (1) gate the NEXT commit on the PREVIOUS round being SETTLED (or expired), not merely committed - otherwise an attacker commits repeatedly without settling, opening many pending randomness windows; (2) store the exact `randomness_account` pubkey committed for the current round in your round state and assert at settle that the account passed in is that same pubkey - otherwise a caller could pass a DIFFERENT (favorable, already-revealed) randomness account at settle. Bind round -> randomness account -> commit slot/time atomically.

- [medium/medium] **Compute-unit and account-ordering coupling between Switchboard ix and your ix in the same tx** — commitIx must execute before close_round_commit in tx1, and revealIx before settle_round in tx2 (the SDK examples place the Switchboard ix first). Your program reads the randomness account that revealIx just updated within the same transaction, so instruction ordering is load-bearing. Also the reveal tx carries the oracle signature verification cost; budget compute units (examples use computeUnitLimitMultiple ~1.3 and a priority fee). This is a client/tx-construction concern but the on-chain instruction must tolerate being the 2nd instruction and must re-derive/validate the account rather than trust position.

**Open questions:**
- Reveal window for a 30s round: do you hard-require reveal in the slot after commit (tutorial's seed_slot == clock.slot-1, very brittle) or accept any time the oracle has revealed within round_secs? This choice drives the entire stuck-round/timeout design and must be decided before coding.
- Is settle_round permissionless (any keeper, recommended to kill selective-abort) or restricted to the committer? The tutorial restricts settle to the user signer; FlipVault's fairness argues for permissionless settle with only a time guard.
- Who pays for and triggers the per-round commit and reveal transactions (founder keeper bot? any user? incentivized?), and what happens economically if nobody bothers to call settle for several rounds?
- On oracle non-reveal/expiry (~1h): is the policy to re-commit the SAME logical round to a fresh slothash, or to cancel the round entirely (no flip that round)? And are deposits made during a stuck round refundable/withdrawable while the round is unsettled?
- Should the round still be 'closable' (flip) if the selected vault has an empty TOKEN-tranche or empty SOL-tranche? This interacts with VRF only insofar as a no-op flip still consumes the round's randomness - confirm a no-op flip is acceptable and doesn't waste the cadence.
- Is one shared, reused Randomness account acceptable, or do you want a fresh Randomness PDA per round? Reuse is cheaper but requires careful state-binding so a stale revealed value from a prior round can never be replayed into a new round (the seed_slot/commit-slot guard must enforce this).

**Recommendations:**
- Adopt crate `switchboard-on-demand = { version = "0.13", features = ["anchor"] }` and type `switchboard_on_demand::accounts::RandomnessAccountData`. Do NOT use switchboard-v2 / the old VRF account+callback model - it is deprecated and is a different program.
- Restructure the single spec `close_round` into TWO instructions: `commit_round` (paired in-tx with `randomness.commitIx(queue)`) and `settle_round` (paired in-tx with `randomness.revealIx()`). Put vault = value[0] % 4 and the flip ONLY in settle_round. Never compute the outcome in commit_round.
- In commit_round: assert Clock now - round.last_settled_time >= round_secs; assert the previous round is settled-or-expired; store round.randomness_account = key, round.commit_slot = clock.slot, round.commit_time = now, round.state = Pending. Optionally assert `get_value(clock.slot).is_err()` to prove not-yet-revealed at commit (anti-peek).
- In settle_round: assert ctx.accounts.randomness_account.key() == round.randomness_account (bind identity); parse with RandomnessAccountData::parse(...); let v = data.get_value(clock.slot).map_err(|_| RandomnessNotResolved)?; let vault = (v[0] % 4) as usize; perform the deterministic sell-first flip; set round.state = Settled, round.last_settled_time = now. Make this instruction PERMISSIONLESS (any signer) guarded only by round state + time, to defeat selective-reveal.
- Decide and document the freshness policy: prefer NOT hard-coding seed_slot == clock.slot-1. Rely on get_value success plus your own guard `now - round.commit_time <= reveal_deadline` (a few slots to ~minutes, well under the ~1h oracle expiry). Reject settle if past your reveal_deadline and route to the timeout path instead.
- Add an explicit timeout/recovery instruction `cancel_or_recommit_round`: callable once now - round.commit_time exceeds reveal_deadline; either (a) cancel the round (no flip, advance last_settled_time so the game continues) or (b) re-commit the same Randomness account to a fresh slothash for the same logical round. This prevents a non-revealing oracle from permanently bricking FlipVault.
- Keep deposit eligibility frozen relative to the committed round: a deposit that arrives between commit and settle should NOT be able to retroactively change which vault flips or front-run the known-at-reveal outcome. Tie eligibility to commit_slot, consistent with the spec's 'immediately eligible if flipped this round' but adjudicated at commit, not reveal.
- For vault selection robustness if vault count ever changes off 4, fold all 32 bytes into a u128 and use rejection sampling instead of a single-byte modulo; for n=4 today, value[0] % 4 is exactly uniform and fine. Always derive selection solely from oracle bytes - never mix in caller-controlled data.
- Minimal VRF account set to plan for: [global round/curve PDA (your state), the reused Randomness account (Switchboard-owned), the Switchboard queue account, the Switchboard On-Demand program, recent_slothashes sysvar (used by commit), Clock sysvar, and the keeper/payer signer]. The commitIx/revealIx are built client-side via the SDK; your instruction only needs read access to the Randomness account plus Clock and your state.
- Write the invariant test (reserve_lamports + Sum(SOL-tranche lamports) == const across flips) to run end-to-end through the real 2-tx commit/reveal on localnet using the Switchboard On-Demand test validator/oracle, so the test exercises the actual reveal timing and the stuck-round/timeout path, not a mocked synchronous callback.

## Dimension: Security & Economic Attack Surface

**Mental model:** FlipVault is a single global constant-product AMM (r_sol real lamports in a reserve PDA, r_tok purely virtual, k = r_sol*r_tok) acting as the house bankroll, plus 4 vaults. Each vault has two tranches; at any instant one tranche is denominated in SOL and the other in virtual TOKEN, and a tranche stores {asset, amount, total_shares}. Users hold non-transferable position PDAs keyed (user, vault, tranche) -> shares. Real lamports only ever live in (a) the reserve PDA or (b) the per-vault PDAs holding the SOL tranche; TOKEN is fictional.

Flow of value: deposit() adds SOL to a vault's SOL tranche and mints pro-rata shares (1:1 if empty); withdraw() burns shares and pays from the SOL tranche; close_round() runs every round_secs, requests Switchboard VRF, and in the settle callback picks vault = rand % 4 and flips it. A flip is sell-first: sell the TOKEN tranche's T tokens into the curve (reserve -> vault lamports), then buy with the SOL tranche's S lamports (vault -> reserve), then swap the two tranches' asset flags while leaving shares untouched. So a flip is a forced two-leg AMM trade against the house reserve, with the *direction of the bet* (whether the vault's SOL holders come out ahead) determined by how the curve price has moved since the position was established.

The economic heart: the reserve is the counterparty. Every flip moves price along the bonding curve and shuttles lamports between reserve and vault. The stated conservation invariant (reserve_lamports + sum of SOL-tranche lamports == constant, changed only by deposit/withdraw) is the solvency backbone. The attack surface is therefore: (1) share accounting (ERC4626-style inflation/donation, rounding), (2) information asymmetry around VRF reveal vs. flip settlement (front-running the known winning vault), (3) VRF integrity (grinding/reroll/bias), (4) reserve solvency under adversarial repeated flips and net-sells, and (5) standard Anchor hygiene (signer/owner checks, overflow, CPI ordering, PDA non-transferability).

**Confirmed:**
- There is exactly ONE global curve shared by all 4 vaults: r_sol (real lamports in a reserve PDA), r_tok (virtual), k = r_sol * r_tok held constant. r_tok and k are pure bookkeeping; only SOL moves as real lamports.
- Buy: tokens_out = r_tok - k/(r_sol+dx), then r_sol += dx, r_tok = k/r_sol. Sell: sol_out = r_sol - k/(r_tok+dy), then r_tok += dy, r_sol = k/r_tok. Standard x*y=k with fee-free quoting.
- A vault has two tranches; at any moment exactly one is SOL-denominated and one is TOKEN-denominated (either may be empty). Each tranche = {asset: SOL|TOKEN, amount, total_shares}.
- deposit(vault, amount): adds SOL only to the vault's current SOL tranche; shares = amount*total_shares/tranche_amount, or 1:1 if empty; lamports go to the vault PDA; deposit is immediately eligible and will flip this round if the vault is selected.
- withdraw(vault, shares): only from a SOL tranche; burns shares, pays shares*tranche_sol/total_shares lamports from the vault PDA. TOKEN-tranche holders cannot withdraw and must wait for the vault to be reselected and flip back to SOL.
- close_round runs at most once per round_secs: requests VRF, then in the settle callback picks vault = rand % 4 and flips that one vault.
- Flip is sequential sell-first: (1) sell the TOKEN tranche's T -> sol_out, reserve->vault; (2) buy with the SOL tranche's S -> tok_out, vault->reserve; (3) swap the two tranches' asset flags; shares untouched.
- Stated solvency invariant: reserve_lamports + sum(SOL-tranche lamports across 4 vaults) is constant except for deposits(+)/withdraws(-); flips never change it, only shuttle lamports between reserve and vaults.
- The spec explicitly flags that the reserve can shrink toward zero on sustained net-sells and that r_sol -> 0 must be guarded.
- Position balances are non-transferable PDAs keyed by (user, vault, tranche) -> shares; there is no SPL token / transferable share.

**Findings:**
- [critical/high] **Deposit/flip front-run after VRF reveal: the winning vault is public before settlement (atomicity gap)** — close_round() requests VRF then flips in a *settle callback*. On Solana the Switchboard On-Demand settle is a separate transaction whose payload (the randomness) is on-chain/derivable the instant it lands. If close_round's settle callback only stores the result (or if vault selection and the flip are not in the SAME instruction as the randomness consumption), then between 'rand revealed' and 'flip executed' the winning vault index (rand % 4) is PUBLIC. An attacker reading the revealed value computes the winner and front-runs the flip with deposit(winner, large_amount), capturing a pro-rata share of a flip whose outcome they already know is favorable (e.g. if the SOL tranche will buy tokens at a price they know is about to be re-sold higher next time the vault flips, or to dodge a losing flip by depositing into the side that wins). Even a 1-slot gap is exploitable via priority-fee ordering. MITIGATION: the randomness MUST be consumed and the flip MUST execute atomically inside the single instruction that reads the VRF result, with no user-callable deposit able to interleave. Bind eligibility to a round_id snapshot taken at close_round *request* time, not settle time: only deposits with round_id < current_round flip. Equivalently, freeze the vault's tranche_amount/total_shares at request time so post-reveal deposits cannot ride the flip. This is the single most important design decision and must be settled before implementation.

- [high/high] **Direct-lamport donation to a Vault PDA inflates share price and steals from the next depositor (ERC4626 first-depositor / inflation attack)** — If tranche_amount used in shares = amount*total_shares/tranche_amount is READ FROM THE VAULT PDA's lamport balance (account.lamports()) rather than from an authoritative in-state counter, an attacker can: (1) be the first depositor with 1 lamport -> mints 1 share (1:1); (2) transfer e.g. 10 SOL directly to the Vault PDA via System Transfer (PDAs can receive lamports with no CPI/owner consent); now share-price = 10_000_000_001 lamports/share; (3) victim deposits 15 SOL -> shares = 15e9 * 1 / 10e9 = 1 (integer floor), so the victim gets 1 share for 15 SOL while attacker's 1 share is now worth ~12.5 SOL — attacker steals ~half. Sample: attacker net +~7.5 SOL. Crucially, even if tranche_amount is tracked in STATE (the correct choice), a donation still corrupts the conservation invariant: reserve + sum(SOL-tranche state amounts) stays constant but the PDA's real lamports now exceed state, and on withdraw the program pays shares*tranche_sol/total_shares from *state* — the donated lamports become stranded/unaccounted, and at flip time step (2) moves S=state-amount (not balance) so the surplus sits in the vault unowned. MITIGATIONS: (a) ALWAYS use in-state tranche.amount, NEVER account.lamports(), for all share math and for S in the flip; (b) enforce a non-trivial minimum first deposit and/or seed each tranche with virtual shares/dead shares (mint initial shares to a burn address, ERC4626 'virtual offset') so 1:1 rounding cannot be weaponized; (c) reconcile/ignore donated surplus explicitly (e.g. sweep to reserve or refuse to count it). Decide state-vs-balance accounting before coding.

- [high/medium] **Rounding direction in mint/withdraw/flip can leak reserve value to users if not uniformly floored against the user** — All three conversions divide and must round in the protocol's favor: (mint) shares minted must round DOWN; (withdraw) lamports paid must round DOWN; (flip) the curve quotes sol_out (sell leg, reserve->vault) must round DOWN and tok_out should round consistently. If shares round UP on deposit OR lamports round UP on withdraw, repeated dust deposits/withdraws extract 1 lamport per op from the reserve/other holders — at scale (cron-able, no per-tx cost beyond fees) this is a slow drain and breaks the invariant. The flip's sell leg is the dangerous one: sol_out = r_sol - k/(r_tok+T) uses integer division of k/(r_tok+T); the integer floor of k/(r_tok+T) should round so that sol_out (what leaves the reserve) is UNDER-stated, i.e. k/(r_tok+T) rounds UP, otherwise the reserve pays out a fraction too much each flip and r_sol drifts down faster than k implies. Because k can be ~1e18 and divisions are u64/u128, the rounding choice is load-bearing. MITIGATION: define a single rule — every division that determines lamports leaving the reserve or shares credited to a user rounds AGAINST the user; add property tests asserting the conservation invariant holds to the lamport across randomized deposit/withdraw/flip sequences. Use u128 (or u256 via checked math) for r_sol*r_tok to avoid the division dominating.

- [high/medium] **VRF grinding / reroll if close_round can be retried or settle is not bound to a one-shot round commit** — Two grinding surfaces: (1) REQUEST grinding — if close_round can be called again before the prior round's settle finalizes (or if a failed/abandoned settle leaves the round re-requestable), the founder/attacker can request VRF, observe the candidate value off-chain (Switchboard reveals are observable), and only let the favorable settle land, abandoning unfavorable ones (re-request = reroll). (2) SETTLE grinding — if the callback re-derives rand from a value that can be influenced (e.g. mixing in a caller-controlled slot/recent-blockhash, or if the settle can be submitted by anyone and reverted by failing a later constraint, the tx can be replayed until rand%4 is the desired vault). The time guard 'once per round_secs' alone is INSUFFICIENT: it limits frequency, not reroll-within-a-round, and a 30s cadence still allows one reroll attempt per 30s indefinitely if abandonment is free. MITIGATIONS: commit-then-reveal with a single live randomness account per round_id; once VRF is requested for round N, store the request and forbid a new request for round N; the settle must be idempotent and may only consume the exact randomness account bound at request time; do not derive the vault index from anything the settler controls (use only the VRF output); ensure an abandoned/failed settle does NOT reset round state to re-requestable (or make re-request cost > expected edge). Also confirm rand%4 has negligible modulo bias (fine for 4 over 256-bit, but document it).

- [high/medium] **Reserve insolvency / drain under sustained adversarial flips and net-sells; r_sol -> 0 guard semantics undefined** — The reserve is the sole counterparty to every flip's sell leg (reserve -> vault). The spec acknowledges r_sol can shrink toward zero. Concrete failure: a flip's sell leg sol_out = r_sol - k/(r_tok+T) can demand more lamports than the reserve PDA actually holds if many vaults are simultaneously TOKEN-heavy and price has run up — the curve PROMISES sol_out but the real reserve may not have it (the invariant says reserve+SOL-tranches is constant, but a single flip can require shuttling nearly all reserve lamports into one vault). If sol_out > reserve_lamports - rent_exempt_minimum, the transfer fails and the flip (and thus close_round) bricks, freezing all rounds. Also as r_sol -> 0, k/(r_sol+dx) and price explode, and r_tok = k/r_sol overflows / divides toward absurd values; the buy leg of a flip with large S could push r_sol arbitrarily high and tok_out tiny, distorting future flips. MITIGATIONS: define explicit behavior when sol_out exceeds available reserve (cap sol_out at reserve-minus-rent and adjust r_sol/r_tok consistently, or skip/partial-flip with documented accounting that preserves the invariant — do NOT let it panic and freeze rounds); enforce a hard floor r_sol >= r_sol_min and reject/clamp trades that would breach it; keep the reserve rent-exempt at all times so the PDA is never closed; stress-test worst-case where rand repeatedly selects the most token-heavy vault.

- [high/medium] **TOKEN-tranche holder early lamport extraction via withdraw on a stale/mislabeled tranche** — withdraw must pay only from a SOL tranche. The risk is a check that keys off the wrong field. After a flip, step (3) swaps the two tranches' asset flags but leaves shares untouched and (per the model) leaves lamports/token amounts in place while relabeling. If withdraw determines 'is this the SOL tranche?' from anything other than the freshly-updated asset flag — e.g. from a cached pointer 'tranche A is always SOL', or from the position PDA's own copy of asset that wasn't updated on flip — a TOKEN holder could call withdraw against the tranche that still has lamports and drain value they shouldn't access (they bet on TOKEN and should be illiquid until reselection). Conversely a SOL holder might be wrongly blocked. Also: the position PDA is keyed by (user, vault, tranche-slot); if tranche identity is a fixed slot index and flips swap *contents* but not slot, the asset flag is the only truth and must be read atomically at withdraw time. MITIGATION: store asset on the tranche state only (single source of truth), have withdraw assert tranche.asset == SOL by reading current state, and ensure flip updates that flag before the round closes. Never store asset on the position PDA. Add a test: deposit, get flipped to TOKEN, attempt withdraw -> must fail; flip back -> withdraw succeeds.

- [high/medium] **Missing signer/owner/PDA constraints, unauthorized close_round/flip, and CPI-ordering/reentrancy on lamport transfers** — Anchor-hygiene cluster that must be specified: (1) close_round / settle callback authority — who may call? If anyone can trigger the VRF request that is acceptable, but the settle that performs the flip must verify the randomness account is the exact one bound to the current round and that round state is 'awaiting settle'; an attacker must not be able to call the flip with a forged randomness account or a different vault index. (2) deposit/withdraw must require the position-owner as signer and derive the position PDA from the signer's key (seeds = [b'pos', user, vault, tranche]) with has_one/owner checks, so user A cannot withdraw against user B's position. (3) Lamport movement: paying out withdraw/flip via direct lamport arithmetic on PDA accounts (not System CPI, since program-owned PDAs adjust .lamports directly) must be ordered AFTER all state mutation and checked-math, and the program must guard against re-entry through any CPI (Switchboard callback) — settle should mark the round settled BEFORE moving lamports/doing the flip so a re-entrant settle can't double-flip. (4) Overflow: r_sol*r_tok must be u128/checked; k/(r_sol+dx) with u64 r_sol and large k overflows u64 — all curve math needs checked u128 and saturating/explicit error on overflow, never silent wrap. (5) Position non-transferability is automatic since shares live in a PDA with no transfer instruction — confirm NO instruction exists that reassigns a position's owner, and that close-account refunds rent only to the deriving owner. MITIGATION: enumerate every account constraint (signer, owner, seeds, has_one, address= for reserve/vault PDAs, the VRF account address pin), use checked_* / u128 throughout, set state-before-transfer ordering, and add the conservation-invariant property test as a CI gate.

- [medium/medium] **Empty-tranche / 1:1 reset edge cases and zero-amount griefing** — When a tranche is empty, deposit mints 1:1. After a flip, a tranche that had its SOL fully bought into TOKEN (S consumed) may relabel with total_shares still > 0 but the *new* SOL side comes from the sold T (sol_out). Edge cases: (a) tranche with total_shares > 0 but amount could momentarily be 0 if T or S rounds to 0 (tiny tranche) -> division by zero in shares = amount*total_shares/tranche_amount or in withdraw payout; (b) depositing amount=0 or shares=0 to grief state / mint zero shares; (c) a flip when a tranche is empty (T=0 -> sol_out=0, or S=0 -> tok_out=0) must be a clean no-op on that leg, not an error that bricks the round; (d) first-ever deposit dust (1 lamport) enabling the inflation attack above. MITIGATION: explicitly handle amount==0 and total_shares==0 separately (1:1 path), reject zero-value deposits/withdraws, guard every divisor (tranche_amount, total_shares, r_tok+T, r_sol+dx) against zero, and define flip behavior when either tranche is empty so close_round never panics.

- [medium/medium] **Founder/initialize trust and parameter griefing** — initialize(seed_sol, init_r_tok, round_secs) lets the founder set the initial price k = seed_sol_curve * init_r_tok and the round cadence. Risks: (1) founder can pick init_r_tok to set an arbitrary starting price; not an exploit per se but front-running depositors with knowledge of the seed matters. (2) If round_secs or other params are mutable post-init by an admin, an admin could stall rounds (freeze TOKEN holders' liquidity) or speed them up. (3) The reserve is 'the house bankroll' and is funded by the founder — if the founder can withdraw the reserve directly (any admin-drain instruction), all vault solvency evaporates. MITIGATION: make round_secs and curve params immutable after init (or behind timelock/multisig), ensure there is NO instruction that lets any authority remove reserve lamports outside the flip mechanics, and document the trust assumption that the seeded reserve is locked as house collateral. Confirm the reserve PDA has no close/withdraw path reachable by a signer.

- [high/medium] **round_secs=30 cadence + public timing enables predictable MEV/sandwich around flips even with hidden vault** — Even with the vault hidden until reveal, the *timing* of close_round is public and the curve is global/shared. A sophisticated actor can pre-position across all 4 vaults (deposit into every SOL tranche) before a round so that whichever vault wins, they hold a share of the favorable side, then withdraw from the losing-direction vaults after settlement — effectively a low-variance arbitrage funded by the reserve if the curve's flip economics are net-positive to depositors on average. Because all 4 vaults share ONE curve, a flip of vault i moves the price that determines the next flip of vault j, so an attacker holding all four sides can harvest the spread the house pays on every sell leg. Sustained, this is the 'reserve drain' vector with a concrete strategy. MITIGATION: model the expected value of a flip to depositors vs. reserve; if flips are EV-positive to depositors (no fee/spread captured by the house), the reserve is structurally guaranteed to bleed and needs an explicit fee/haircut on flips or on withdraw to make the house EV-neutral-or-positive. This is an economic-design decision that must precede implementation.

**Open questions:**
- ATOMICITY: Does the Switchboard settle callback consume the randomness AND perform the flip in a single instruction, or does settle merely store rand for a later flip tx? This determines whether the winning vault is publicly knowable before the flip (the critical front-run). Must be specified.
- ELIGIBILITY SNAPSHOT: Is a deposit 'eligible to flip this round' based on the round state at close_round REQUEST time or at SETTLE time? The spec says 'immediately eligible' — does that mean a deposit landing after VRF reveal but before settle still flips? It must NOT.
- ACCOUNTING SOURCE: Is tranche 'amount' (and S in the flip) read from in-state counters or from the Vault PDA's actual lamport balance? This decides the donation/inflation attack surface.
- ROUNDING POLICY: What is the canonical rounding direction for (a) share mint, (b) withdraw payout, (c) flip sol_out and tok_out? Are all divisions floored against the user, and is curve math in u128/u256?
- RESERVE SHORTFALL: What happens when a flip's sell-leg sol_out exceeds the reserve's available (rent-exempt-adjusted) lamports? Cap, skip, partial, or panic? Undefined behavior here can freeze all rounds.
- r_sol FLOOR: What is the concrete r_sol_min guard value and what does the program do when a trade would breach it — reject, clamp, or pause flips?
- HOUSE EDGE: Are flips EV-neutral/positive for the house, or do depositors extract expected value from the reserve each flip? Is there any fee/spread/haircut? Without one, sustained net activity drains the reserve by design.
- REROLL: Can close_round be re-requested within a round if a settle fails or is abandoned, and can the settler influence the value mixed into the outcome? Defines the VRF-grinding surface.
- ADMIN POWERS: Are round_secs and curve params mutable post-init, and is there ANY instruction by which an authority can remove lamports from the reserve PDA outside flip mechanics?
- AUTHORITY: Who is authorized to call close_round / its settle, and how is the bound randomness account pinned (address constraint) so a forged or stale VRF account cannot drive a chosen-vault flip?
- EMPTY/ZERO TRANCHE: Defined behavior for flips when a tranche is empty (T=0 or S=0), and for deposits/withdraws when tranche_amount or total_shares is 0, to avoid div-by-zero and round-bricking panics.

**Recommendations:**
- DESIGN THE VRF->FLIP PATH AS A SINGLE ATOMIC, NON-INTERLEAVABLE INSTRUCTION. Bind a round_id at close_round request; freeze each vault's eligible (amount, total_shares) snapshot at request time; only deposits with round_id strictly less than the settling round may participate in the flip. Reject or no-op the flip if the bound randomness account doesn't match the round's committed request. This kills the post-reveal deposit front-run (top risk).
- USE IN-STATE tranche.amount FOR ALL SHARE MATH AND FOR S IN THE FLIP — NEVER account.lamports(). Add a reconciliation/sweep rule for any donated surplus (route stray lamports to the reserve or refuse to count them) so direct lamport donation cannot inflate share price or strand funds. Seed each tranche with virtual/dead shares (ERC4626 virtual-offset) and enforce a minimum first deposit to neutralize 1:1 rounding abuse.
- ESTABLISH ONE ROUNDING LAW: every division that credits shares to a user or pays lamports out of the reserve/vault rounds AGAINST the user (floor user gains, ceil what the user owes / under-state what the reserve pays). Do all curve math in u128 (checked), with explicit overflow errors, and add randomized property tests asserting reserve_lamports + sum(SOL-tranche state amounts) is invariant to the lamport across thousands of deposit/withdraw/flip sequences.
- MAKE THE HOUSE EV-NEUTRAL-OR-POSITIVE: introduce an explicit spread/fee/haircut on flips or withdrawals sized to cover the curve slippage the reserve pays on each sell leg; otherwise model proves the reserve bleeds under sustained activity (and the pre-position-all-4-vaults strategy harvests it). Decide and bake this into the curve math before coding.
- DEFINE RESERVE-SHORTFALL AND r_sol FLOOR SEMANTICS EXPLICITLY so close_round can never panic and freeze rounds: keep the reserve rent-exempt always; if a flip's sol_out would exceed available reserve or breach r_sol_min, clamp/partial-flip with documented invariant-preserving accounting rather than reverting the whole round.
- HARDEN VRF INTEGRITY: one randomness account per round_id, request stored and non-re-requestable for that round; settle idempotent and address-pinned to that account; derive vault index ONLY from VRF output (no settler-controlled inputs); ensure a failed/abandoned settle does not reset the round to re-requestable. Document rand%4 modulo bias as negligible.
- SPECIFY EVERY ANCHOR CONSTRAINT UP FRONT: deposit/withdraw require the position owner as signer with seeds-derived position PDA and has_one/owner checks; reserve and vault PDAs pinned by address=; settle pins the VRF account by address=; mark round 'settled' and mutate all state BEFORE moving any lamports (state-before-transfer) to block re-entrant double-flip via the callback; confirm no instruction reassigns position ownership and no path drains the reserve outside flips.
- DERIVE 'is SOL tranche' SOLELY FROM THE CURRENT tranche.asset FLAG updated by the flip; never cache asset on the position PDA. Add explicit tests: deposit -> get flipped to TOKEN -> withdraw must fail; flip back -> withdraw succeeds; and TOKEN holder can never touch lamports early.
- HANDLE EMPTY/ZERO EDGE CASES EXPLICITLY: separate 1:1 path for total_shares==0; reject zero-value deposits/withdraws; guard every divisor (tranche_amount, total_shares, r_tok+T, r_sol+dx, r_tok) against zero; make empty-tranche flip legs (T=0 or S=0) clean no-ops so the round never bricks.
- LOCK FOUNDER/ADMIN POWERS: make round_secs and curve params immutable after init (or timelocked/multisig), and guarantee there is no reachable instruction that removes reserve lamports outside flip mechanics; document the trust assumption that the seeded reserve is permanently locked house collateral.
- RANK FOR DESIGN-BEFORE-CODE: (1) atomic VRF->flip / post-reveal deposit front-run [critical]; (2) house EV / reserve-drain economics including pre-position-all-4 strategy [high]; (3) state-vs-balance accounting + donation/inflation [high]; (4) rounding-direction leakage with invariant tests [high]; (5) reserve-shortfall & r_sol floor non-panic behavior [high]; (6) VRF grinding/reroll prevention [high]; (7) withdraw-only-from-SOL enforcement via live asset flag [high]; (8) Anchor signer/owner/CPI-ordering hygiene [high]. Resolve 1-6 before writing the curve/round modules.

## Dimension: Edge cases, ambiguities & open spec questions

**Mental model:** FlipVault is a single shared constant-product AMM (one virtual token reserve r_tok, one real SOL reserve r_sol, k = r_sol*r_tok) that the program trades against ITSELF on behalf of 4 vaults. There is no user-facing buy/sell and no SPL token. The only thing users do is deposit SOL into / withdraw SOL from whichever tranche of a vault is currently SOL-denominated. The gamble: each vault has two tranche "slots"; at any instant one slot is SOL-denominated and the other TOKEN-denominated. Once per round (~30s), close_round draws VRF, picks one of 4 vaults (rand % 4), and "flips" it: the program sells the vault's TOKEN tranche back into the curve for SOL (lamports move reserve->vault), buys virtual tokens with the vault's SOL tranche (lamports move vault->reserve), then swaps the two slots' asset flags. Shares are untouched, so a slot's holders keep their pro-rata fraction but their denomination (and therefore their realized SOL value at the next flip) changed at curve prices. A user who is in the TOKEN tranche is LOCKED — they cannot withdraw until their vault is selected again and that tranche flips back to SOL, at an unknown future price. The curve reserve r_sol doubles as the house bankroll that funds the sol_out leg of every flip; the spec's global invariant is reserve_lamports + sum(SOL-tranche lamports) == const except for deposits(+)/withdrawals(-). My read is that flips are the ONLY trading event, the token has no decimals defined, and several genesis/empty/timing behaviors are left unstated and must be pinned before coding.

**Confirmed:**
- No SPL mint exists; the game token is purely virtual — a u64 reserve r_tok inside the curve. Only SOL moves as real lamports.
- There is ONE global curve (r_sol real lamports in a reserve PDA, r_tok virtual, k=r_sol*r_tok) shared across all 4 vaults.
- Each vault has exactly two tranches; at any moment one is SOL-denominated and the other TOKEN-denominated (either may be empty). A tranche stores {asset, amount, total_shares}.
- Per-user balances are non-transferable position PDAs keyed user x vault x tranche -> shares.
- Deposit adds SOL to the vault's CURRENT SOL tranche, minting shares pro-rata (amount*total_shares/tranche_amount, or 1:1 if empty); lamports go to the vault PDA and the deposit is immediately flip-eligible this round.
- Withdraw is allowed ONLY from a SOL tranche: burn shares, pay shares*tranche_sol/total_shares lamports out of the vault PDA. TOKEN-tranche holders cannot withdraw and must wait for reselection.
- Flip is sequential, sell-first: (1) sell the whole TOKEN tranche T into the curve for sol_out, moving sol_out reserve->vault; (2) buy with the whole SOL tranche S, moving S vault->reserve; (3) swap the two tranches' asset flags, leaving shares untouched.
- close_round runs at most once per round_secs (default 30): request VRF, then in the settle callback compute vault = rand % 4 and flip exactly that one vault.
- initialize funds the reserve PDA with seed_sol and sets r_tok and k; the reserve is the house bankroll.
- The intended global invariant: reserve_lamports + sum over 4 vaults of SOL-tranche lamports is constant, changed only by deposits(+) and withdrawals(-), never by flips; flips only shuttle lamports between reserve and vaults.
- r_sol can shrink toward zero on sustained net sells and must be guarded against r_sol -> 0.

**Findings:**
- [high/high] **Genesis tranche layout is unspecified: which slot is SOL vs TOKEN, and is the TOKEN slot empty or nonexistent at init?** — initialize() sets the curve but says nothing about the 4 vaults' initial tranche state. Open decisions: (a) Does each vault start with slot A = SOL tranche (amount 0, total_shares 0) and slot B = TOKEN tranche (amount 0, total_shares 0)? (b) Is the asset flag stored per-slot, and which physical slot index is SOL at genesis? This matters because deposit() must target 'the current SOL tranche' — code needs a deterministic rule to find it. If both slots start empty and SOL-flagged, or if the SOL/TOKEN flag is undefined at genesis, deposit() has no well-defined target. Recommend: all 4 vaults init with slot0=SOL(0,0), slot1=TOKEN(0,0). Also note the first flip of a fresh vault is a double no-op (T=0 and S=0) and merely swaps two empty slots — a wasted round (see separate finding).

- [high/high] **Empty-tranche flips (T=0 and/or S=0) have undefined behavior and create a wasted-round / locked-funds class of bugs** — Flip step 1 sells T; if T=0 then sol_out = r_sol - k/(r_tok+0) = 0, a no-op. Step 2 buys with S; if S=0 then tok_out=0, a no-op. Four cases: (i) S>0,T=0: the SOL tranche buys tokens (locking those depositors) and the empty token tranche flips to an empty SOL tranche — normal. (ii) S=0,T>0: token tranche sells to SOL (unlocking holders) and empty SOL flips to empty token — normal, the 'good' outcome. (iii) S=0,T=0: pure no-op, an entirely wasted round that consumed a VRF call and the 30s window — a fresh/idle vault being picked burns a round for everyone. (iv) S>0,T>0: both legs execute. The spec must state that all four are legal and that case (iii) is an accepted wasted round (or that close_round should re-roll / pick a non-empty vault). Also: must flips with a 0 leg still emit events / advance round state? Recommend documenting all four explicitly and NOT re-rolling (re-rolling biases VRF and complicates the callback).

- [high/high] **A user can hold BOTH tranches of one vault, but the spec never confirms it nor defines whether withdraw/deposit is per-slot — concurrency of the two position PDAs needs to be explicit** — Positions are keyed user x vault x tranche, so a user CAN hold shares in slot0 and slot1 of the same vault simultaneously (e.g., deposit, get flipped into TOKEN, then deposit again into the now-SOL other slot). This is almost certainly intended but unstated; confirm it. Critical sub-question: is the position PDA seed keyed by PHYSICAL slot index (0/1) or by ASSET flag (SOL/TOKEN)? It MUST be physical slot index, because the asset flag flips while shares persist — keying by asset flag would corrupt ledgers on every flip. The spec's phrase 'a tranche slot's share ledger persists across flips while its asset flag flips' implies physical-slot keying; make this a hard requirement, or the whole shares-untouched model breaks.

- [medium/high] **No standalone buy/sell exists — confirm trading happens ONLY via flips, and that this is the intended lock mechanic** — Nothing in Instructions exposes user-initiated buy or sell against the curve. The only curve interactions are internal to flip(). Consequence: a depositor who lands in a SOL tranche that then gets flipped is converted to TOKEN at the current curve price with no opt-out, and is locked until their vault is reselected AND that slot flips back to SOL — an unbounded wait at an unknown future price. This is the core gamble and should be stated as a deliberate invariant, not left implicit, because an implementer might 'helpfully' add a redeem path that breaks the bankroll invariant. Confirm: there is intentionally no way to exit a TOKEN tranche except a future flip.

- [critical/high] **Virtual-token units/decimals and lamport scale are undefined, risking integer-truncation and overflow in k = r_sol * r_tok** — r_sol is in lamports (1 SOL = 1e9). r_tok has no defined decimals. k = r_sol*r_tok is computed in integers: if r_sol ~ 1e9 lamports per SOL and a bankroll of, say, 100 SOL = 1e11, and r_tok is also ~1e11, then k ~ 1e22 which overflows u64 (max ~1.8e19). This FORCES u128 (or u256) for k and all intermediate products, and a decided fixed-point convention for r_tok. Also every division (k/(r_sol+dx), shares math, payout math) truncates toward zero — rounding direction must be chosen to always favor the reserve/protocol (round token_out and sol_out DOWN, round shares burned semantics carefully) to prevent draining via rounding. Decide: r_tok decimals, k width (u128 mandatory), and a documented rounding policy. Provide a concrete genesis example, e.g. seed_sol=100 SOL (1e11 lamports), init_r_tok=1e11 -> price ~1 token/lamport, k=1e22 (u128).

- [medium/high] **Same vault can be picked on consecutive rounds; spec doesn't say whether that's allowed or mitigated, and 'rand % 4' biases toward fairness only if rand is uniform over a multiple of 4** — vault = rand % 4 with uniform 256-bit VRF output is effectively unbiased (4 divides 2^256 negligibly off), so modulo bias is not a real concern here — note that for the record. But the GAMEPLAY question is open: can the same vault flip twice in a row (or repeatedly)? Nothing prevents it. Repeated selection of the same vault rapidly cycles its two slots (SOL->TOKEN->SOL...) and can churn depositors and drain/refill the reserve faster than other vaults ever move. Decide whether repeats are allowed (simplest, recommend yes) or excluded (needs last_picked state and rand % 3 over the others). Also confirm the 4 vaults are otherwise identical (no per-vault parameters).

- [high/high] **close_round timing/liveness: 'once per round_secs' lower-bound vs late calls, who can call it, and the VRF request/settle gap are underspecified** — Open questions: (1) Is round_secs a MINIMUM spacing (>= last_round_ts + round_secs) or a fixed schedule? If a keeper calls late (e.g., 90s after), does one flip happen or do missed rounds accumulate/catch up? Recommend: single flip per call, gated by 'now >= next_round_ts', then next_round_ts = now + round_secs (no catch-up). (2) Who is authorized to call close_round — permissionless keeper, or authority-only? Permissionless is more live but lets an attacker time deposits around it. (3) The VRF is two-phase (request in close_round, pick+flip in the settle callback). Between request and settle, what is the round's state? Can deposits/withdrawals still occur? Can a second close_round be requested while one is pending? Must define a 'round pending' lock and decide whether deposits/withdrawals are frozen during the pending window — this is the main MEV/race surface.

- [high/high] **Deposit/withdraw vs close_round/settle ordering is a live race that determines fairness and can be gamed** — Spec says a deposit is 'immediately eligible — if the vault is flipped this round, it flips.' But with two-phase VRF there are two distinct moments: the close_round request and the settle callback. If deposits are allowed AFTER the VRF is requested but BEFORE settle, a sophisticated actor could observe pending randomness state (or the request slot) and deposit/withdraw to dodge or chase a flip, or worse, if VRF result is partially predictable, front-run it. Conversely, withdraw mid-round from a SOL tranche right before its vault flips lets a user exit at book value and avoid the gamble. Decide: (a) freeze deposits/withdrawals on a vault (or globally) once a round is pending until settle completes; or (b) snapshot eligibility at request time. Without a stated rule, the implementation will be exploitable. Also define same-round deposit-then-withdraw (allowed? both from SOL tranche, net zero minus rent).

- [medium/high] **Withdraw from a SOL tranche immediately after a flip lets holders realize favorable conversions risk-free, draining the vault relative to shares** — After a flip, the slot that just became SOL holds sol_out (proceeds of selling the old TOKEN tranche). Its share-holders (who were locked as TOKEN) can now withdraw shares*tranche_sol/total_shares. If the curve moved favorably, they exit with profit; if unfavorably, they're underwater but can still exit. This is intended, but the open question is timing relative to close_round: can they withdraw in the SAME round/slot the flip settled, before any further deposits dilute or further flips occur? And does withdraw need any guard so that a withdraw can't make tranche_amount/total_shares math divide-by-zero (last withdrawer empties the tranche -> total_shares=0; next deposit must hit the 1:1 empty branch). Confirm: full-withdraw resets the tranche to (amount 0, total_shares 0) and re-arms the 1:1 mint path.

- [critical/high] **House-bankroll insolvency (r_sol -> 0) semantics for users are undefined: what fails, and are vault lamports ever at risk?** — The reserve funds the sol_out leg of every flip (reserve->vault). On sustained net sells, r_sol shrinks. Open questions: (1) What is the guard exactly — reject the flip? clamp sol_out to available reserve lamports? cap r_sol at a floor? Each choice changes the constant-product math and could break k. (2) If a flip's sol_out exceeds reserve lamports, the move reserve->vault would fail — does the whole close_round revert, leaving the round un-flipped (and the vault stuck)? (3) Does insolvency ever touch user principal? Per the invariant, vault SOL-tranche lamports are segregated from the reserve, so withdrawals from SOL tranches should remain honored even if the reserve is empty; CONFIRM that user-deposited lamports in vault PDAs are never used to cover flip shortfalls. (4) Note r_sol can mathematically never reach exactly 0 via constant-product (k/(r_tok+dy) > 0 always), but it can round to 0 or fall below a dust floor — define the floor. This is the single most important solvency decision and is only gestured at ('guard against r_sol -> 0').

- [high/medium] **Rent-exemption and lamport accounting interact with the invariant — the 'vault PDA lamports' are not purely tranche SOL** — Every PDA (reserve, 4 vaults, every position) must stay rent-exempt. The invariant 'reserve_lamports + sum(SOL-tranche lamports) == const' must be defined over the TRACKED tranche.amount fields, NOT raw account lamports, because raw lamports include rent and possibly the other (TOKEN) tranche's residual. Also: a vault PDA holds lamports for BOTH its tranches' bookkeeping but only the SOL tranche has real lamports; when slots swap on flip, the physical lamports in the vault PDA change (sol_out came in, S left) — ensure tranche.amount (tracked) and the PDA's spendable lamports stay reconciled, net of rent. Decide whether reserve and vault PDAs are system-owned (lamport-only) or program-data accounts, since CPI lamport transfers out of a program-owned PDA require direct lamport mutation, not system transfer.

- [medium/high] **Authority/upgrade powers and trust assumptions are unspecified beyond 'founder funds reserve'** — Open: (1) Is there an admin authority after initialize, and what can it do (pause, change round_secs, set VRF config, drain reserve, close vaults)? Any ability to withdraw the reserve = rug risk for token-tranche holders whose future payout depends on reserve solvency. (2) Program upgrade authority — if upgradeable, the deployer can change flip math arbitrarily; state whether it will be burned/multisig. (3) Can the founder reclaim seed_sol? If the reserve is 'house bankroll,' founder economics (do they take a cut of flips? of the spread?) are entirely undefined — as written, the house has edge only via curve mechanics and rounding, not an explicit fee. Decide the fee model (currently none) and the admin trust model explicitly, because users are locked and depend on it.

- [medium/medium] **Pro-rata share math has a first-depositor / dust-inflation edge and an unstated precision for the 1:1 empty branch** — shares = amount*total_shares/tranche_amount truncates; with small total_shares and large tranche_amount this can yield 0 shares for a real deposit (donation attack / dust grief), and the classic first-depositor inflation attack applies if someone can donate lamports directly to a vault PDA to skew tranche_amount before total_shares grows. Because lamports can be sent to a PDA out-of-band, tranche_amount derived from raw balance would be manipulable — another reason to track tranche.amount as an explicit field updated only by deposit/withdraw/flip, never read from account balance. Decide: reject deposits that would mint 0 shares; consider a minimum first deposit; and define share decimals/precision (u64 vs u128 for total_shares).

- [medium/medium] **Flip step ordering uses r_tok/r_sol values that change between the sell and buy legs — confirm the intended sequential (not simultaneous) pricing** — The spec is explicit that flip is sequential sell-first: step 1 mutates r_sol/r_tok (sell T), then step 2 prices the buy of S against the ALREADY-UPDATED reserves. This is intentional and means the buy gets a different price than if done simultaneously. Worth confirming because it has economic consequences: a vault with large T then large S effectively sells low and buys high within one flip (slippage against itself), and the order (sell-first vs buy-first) is a design choice that changes payouts. Provide a worked example in tests. Edge: if both S and T are large relative to reserves, the sequential moves can push r_sol very low on... wait — selling T RAISES r_sol (sol_out leaves reserve... no: sell moves reserve->vault, lowering reserve r_sol? The spec's curve sell does r_sol -= sol_out). So step 1 (sell T) DECREASES r_sol, increasing insolvency risk before step 2 buys it back. Ordering therefore directly affects the r_sol->0 guard. Confirm sell-first is intended given it stresses the reserve more than buy-first.

**Open questions:**
- Genesis: what are the exact initial values of each vault's two tranches (asset flag per physical slot, amount=0, total_shares=0)? Which physical slot is SOL at init?
- Are position PDAs keyed by physical slot index (required for ledger persistence across flips) or by asset flag (would corrupt on flip)? Confirm physical-slot keying.
- Empty-tranche flips: are all four (S/T zero combinations) legal, and is an all-empty flip an accepted wasted round or should close_round re-roll / skip to a non-empty vault?
- Is there truly NO user buy/sell — trading only via flips — and is the TOKEN-tranche lock with no exit until reselection the intended, permanent mechanic?
- What are the virtual token's decimals/units, and is k computed and stored as u128 (or wider)? What is the documented rounding direction for every division?
- Is round_secs a minimum spacing or a fixed cadence; do missed/late rounds accumulate (catch-up) or collapse to a single flip per call?
- Who may call close_round — permissionless keeper or authority-only?
- During the VRF request->settle window, are deposits and withdrawals frozen (per-vault or global), and can a second round be requested? What exactly is snapshotted as flip-eligible (state at request vs at settle)?
- Can a user withdraw from a SOL tranche mid-round before its vault flips (dodging the gamble), and same-round deposit-then-withdraw — allowed?
- Can the same vault be selected on consecutive rounds, or must selection exclude the last-picked vault?
- Exact r_sol->0 guard: reject flip, clamp sol_out to reserve, or floor r_sol — and does a flip shortfall revert the whole close_round, leaving the vault stuck? Define the dust floor.
- Is user principal in vault SOL tranches ever used to cover reserve shortfalls, or strictly segregated (withdrawals always honored regardless of reserve solvency)?
- Is the invariant defined over tracked tranche.amount fields (net of rent) rather than raw PDA lamports? Are reserve/vault PDAs system-owned or program-owned (affects how lamports are moved)?
- Admin/authority powers post-init (pause, change round_secs, drain reserve), upgrade-authority disposition (burned/multisig?), and can the founder reclaim seed_sol?
- Is there any protocol/founder fee, or is the house edge purely curve slippage + rounding? Define the fee model (currently none).
- First-depositor/dust: reject deposits that would mint 0 shares? Minimum first deposit? Share precision (u64 vs u128 for total_shares)?
- Is sell-first ordering intentional given it lowers r_sol before the buy leg, maximizing reserve stress and insolvency risk within a single flip?

**Recommendations:**
- Pin genesis explicitly: initialize() creates all 4 vaults with physical slot0 = {asset: SOL, amount: 0, total_shares: 0} and slot1 = {asset: TOKEN, amount: 0, total_shares: 0}. Store the asset flag per physical slot; deposit() resolves 'current SOL tranche' by scanning the two slots for the SOL flag.
- Make position PDA seeds = [b"pos", user, vault_id, physical_slot_index]; NEVER key by asset flag. Document that the asset flag mutates but the slot's share ledger is immutable across flips.
- Use u128 for k and all intermediate products; pick explicit r_tok decimals (suggest matching lamports at 1e9, or define a separate TOKEN_DECIMALS constant). Provide a genesis worked example and assert k fits u128.
- Adopt a single global rounding policy: token_out and sol_out round DOWN (favor reserve); withdrawals round payout DOWN (favor remaining holders); reject any deposit that would mint 0 shares. Add unit tests asserting the protocol never loses lamports to rounding.
- Track tranche.amount, total_shares, and r_sol/r_tok as explicit on-chain fields mutated ONLY by deposit/withdraw/flip — never derive economic amounts from raw account balances (defeats donation/inflation attacks). Reconcile tracked amounts vs spendable lamports (net of rent) in an invariant test.
- Define close_round as: require now >= next_round_ts; set a round_pending lock; request VRF; in settle, pick vault = (rand % 4), flip, clear lock, set next_round_ts = now + round_secs with NO catch-up. Decide and document permissioned vs permissionless caller.
- Freeze deposits and withdrawals (at minimum on the selected vault; simplest is globally) while round_pending is set, so eligibility cannot be gamed between VRF request and settle. State that flip-eligibility is snapshotted at request time.
- Specify the insolvency guard precisely: before moving sol_out reserve->vault, require reserve_lamports - sol_out >= rent_floor + dust_floor; if not, revert the flip (and the whole close_round) and surface an event, leaving r_sol intact. Guarantee in code and tests that vault SOL-tranche withdrawals are always payable from segregated vault PDAs regardless of reserve state.
- Allow the same vault to be reselected (simplest, unbiased) unless gameplay requires fairness rotation; if rotation is desired, store last_picked and draw over the remaining 3. Document the choice.
- Decide the fee/edge model up front: if the house needs guaranteed edge, add an explicit flip fee skimmed to the reserve (round in protocol's favor); otherwise document that edge is purely slippage+rounding and that the reserve can be depleted, exposing token-tranche holders to insolvency risk.
- Decide and document admin trust: ideally no reserve-drain authority post-init, round_secs/VRF config changeable only by a timelock/multisig, and program upgrade authority burned or multisig'd. If founder can reclaim seed_sol, gate it so it cannot break the bankroll invariant for locked holders.
- Write the invariant test first (TDD): assert reserve + sum(SOL tranche amounts) is constant across a randomized sequence of flips, and changes by exactly the signed sum of deposits/withdrawals — including empty-tranche, repeated-vault, and near-insolvency scenarios.
