### v-latency — **partial** (conf: medium)

An Arcium confidential computation completes reliably within a ~30-second window, so a once-per-30-seconds confidential "flip" computation is performance-feasible.

**Evidence:**
- TYPICAL LATENCY IS SECONDS, NOT TENS OF SECONDS (strongest, real-world): Umbra — a production confidential-transfer app running on Arcium mainnet — documents that 'Umbra MPC operations take a few seconds longer than a standard token transfer — there is an off-chain MPC round-trip between the two on-chain instructions,' and that the SDK 'waits for both transactions to confirm before returning.' Pure-ZK paths that skip the MPC round-trip 'finish in normal Solana confirmation time.' This is the most concrete UX latency datum and puts a single confidential op comfortably in the single-digit-seconds range, far inside 30s. (https://sdk.umbraprivacy.com/concepts/how-umbra-works)
- PER-REQUEST OVERHEAD IS ~1s, NOT TENS OF SECONDS: Arcium's 'Shared Rescue Keys / Caching Optimization' writeup states that each encrypted instruction previously required recomputing a Diffie-Hellman-derived shared secret in MPC, 'adding roughly 1 second of overhead per request,' now eliminated via local caching after the first interaction. Confirms baseline crypto overhead is on the order of ~1s. (https://www.arcium.com/articles/shared-rescue-keys-caching-optimization, via WebSearch snippet)
- THROUGHPUT BOUNDS THE PER-OP COST: Arcium/Cerberus is reported '10,000 to 30,000 times faster than FHE'; an MXE can run '~100 computations per second' for simple ops and '~5 per second' for complex ops per cluster. A constant-product buy/sell + P&L + balance update is a small fixed-point arithmetic circuit (simple end of the spectrum), so per-op compute is sub-second to low-seconds. (https://www.decentralised.co/p/the-end-of-just-trust-us)
- THE COMPUTATION IS WELL-SHAPED FOR ARCIUM: the flip math is a small, fixed-shape arithmetic circuit, which fits Arcium's Arcis constraints (no Vec/String/HashMap, no while/loop, fixed-bound for-loops, single exit path) and the ~1232-byte callback output limit — so it is the kind of cheap computation that finalizes fast, not a heavy ML workload. (https://docs.arcium.com/developers/limitations)
- PRODUCTION DEPLOYMENT EXISTS AND IS BEING USED: Arcium Mainnet Alpha launched Feb 2026 and has processed 900k+ computations / 3.5M+ transactions, and the two-tx handler+callback lifecycle is the documented, working pattern (awaitComputationFinalization at 'confirmed' commitment). So sub-30s confidential ops are demonstrably happening in production today. (https://docs.arcium.com/developers/computation-lifecycle ; https://ts.arcium.com/docs)

**Caveats:**
- 'RELIABLY' IS UNSUPPORTED BY ANY PUBLISHED SLA: Arcium's docs publish NO maximum computation duration, latency SLA, finalization-time guarantee, or timeout. The only latency descriptor anywhere is Umbra's informal 'a few seconds.' There is no documented worst-case or tail-latency figure, so a hard '<30s every single time' guarantee is not backed by evidence. (docs.arcium.com/developers/limitations, /computations/computation-tasks)
- ALPHA-STAGE, PERMISSIONED NETWORK = NO LIVENESS GUARANTEE: Mainnet is explicitly 'Alpha,' a 'permissioned MPC network' with a 'curated set of compute nodes,' 'focused on reliability and usability rather than full decentralization.' Early-stage infra can have node downtime, scheduling/queue delays, and degraded latency under load — none of which is documented or guaranteed. (Messari mainnet-alpha report; The Block, https://www.theblock.co/post/387564/...)
- LATENCY = MPC ROUND-TRIP + TWO SOLANA CONFIRMATIONS, NOT JUST COMPUTE: each confidential op is two on-chain txs (handler queues; callback posts result). End-to-end time includes Solana confirmation of BOTH legs plus the off-chain MPC round-trip and Arcium scheduling. Solana confirmation alone is variable and can spike during congestion; that variance stacks on top of MPC time and is outside Arcium's control.
- THROUGHPUT IS PER-CLUSTER AND CONTENTION-SENSITIVE: '~100/s simple, ~5/s complex' is shared cluster capacity. If many MXEs or apps contend for the same cluster, your flip can queue. The game's strict fixed 30s cadence means a single slow/queued/failed computation directly blocks the next round unless you design for it. No public data on contention behavior under real load.
- NO INDEPENDENT BENCHMARK FOUND: every concrete number traces to Arcium/Umbra's own marketing/docs or secondary writeups citing them. I found NO independent third-party measured benchmark of end-to-end confidential-computation latency on Arcium mainnet alpha. Treat the 10,000-30,000x and 100/s figures as vendor claims (preprocessing excluded from the speedup framing). Confidence in the 'seconds' figure rests mainly on Umbra's production docs.
- PREPROCESSING DEPENDENCY: Cerberus uses an offline preprocessing phase (multiplication triples) consumed by the online phase. If preprocessing material is exhausted under sustained 2-flips-per-minute load, the online phase could stall waiting for replenishment. Docs do not state whether preprocessing can keep up with a fixed cadence or what happens when it can't.

**Fallback:** Treat the 30s window as a SOFT cadence, not a hard real-time deadline, and decouple selection from settlement. Concrete design: (1) Keep the VRF selection public/verifiable and on its own 30s tick (this never depends on Arcium). (2) Fire the confidential flip computation as an ASYNC job keyed to the selected box; do NOT block the next tick on its finalization. Settle the flip when awaitComputationFinalization returns (empirically a few seconds), and tolerate occasional spillover past 30s by queueing — the curve state update is serialized by the on-chain Arcium callback ordering, so a late flip still applies correctly. (3) Add an explicit timeout + retry: if a computation has not finalized within e.g. 25-28s, retry/resubmit; if it repeatedly fails, fall back to a 'skip this round' or 'roll the selection into the next tick' policy so the game never deadlocks. (4) PILOT-MEASURE before committing: deploy the flip circuit to Arcium devnet/testnet and measure real end-to-end p50/p95/p99 finalization latency over hundreds of runs under simulated load — do not trust the 'a few seconds' figure for a production cadence without your own numbers. (5) If measured p99 exceeds the window or the alpha network proves flaky, either lengthen the tick (e.g. 60-90s) or run the flip math in a TEE/your-own-MPC path with deterministic latency. Also validate that the buy/sell+P&L circuit fits Arcis fixed-shape constraints and the ~1232-byte output limit during the pilot.

## Research dimensions


### Arcium Overview + Current Status (2026): maturity, model, terminology, and production-readiness for FlipVault Path B (overview)

Arcium is a Solana-native (chain-agnostic-leaning) decentralized confidential-computing network: developers write confidential logic in Arcis (a Rust DSL), it compiles to circuits executed by clusters of "Arx nodes" inside MXEs (Multi-party eXecution Environments) using MPC, and results return to a normal Anchor program via async callbacks. The model is "mechanism public, data private" — exactly the framing Path B wants. As of mid-2026 it is REAL and reachable, but NOT yet fully mature/decentralized: a public testnet launched May 2025 (~3,000 nodes), Mainnet Alpha went live Feb 2026 ("with some centralized components / training wheels for security and stress testing"), the $ARX token launched ~June 22 2026 (~$400M FDV), and the network crossed 1M+ confidential computations (~200k/day) by June 2026. Real teams are building (Zinc, Umbra, Streamflow, Crafts, Dinaro, Bench, Melee, etc.). The toolchain (arcup/arcium init/build/test/localnet/deploy, @arcium-hq/client TS SDK) and devnet+mainnet cluster offsets are documented and appear permissionless. NET: production-usable-but-early — usable for a devnet/testnet build now and a Mainnet-Alpha launch with caveats, but it is alpha infrastructure (partial centralization, async-callback latency on the order of ~1s+ per computation plus Solana confirmation, undocumented hard throughput/cost numbers, audit details not publicly enumerated). Custody-privacy (hiding deposit/withdraw amounts) likely depends on Confidential SPL (C-SPL), which is roadmapped to Devnet in a "Phase 2" and is the least-confirmed piece. Several specific numbers (exact node count in alpha, per-computation cost, named audit firms, precise decentralization date) are stale/unconfirmed and flagged below.

**Findings:**
- [high] What Arcium is: A decentralized confidential-computing network where MPC nodes ('Arx nodes') compute over ENCRYPTED data so neither the nodes nor observers see plaintext. Positioned as an 'encrypted supercomputer' for Solana (and broader web3/web2). Core value prop matches Path B exactly: keep the mechanism (selection/VRF) public while the data (balances, positions, P&L, curve state) stays encrypted and is operated on confidentially.
- [high] Execution model (MXE + clusters + Arx nodes): An MXE (Multi-party eXecution Environment) is a configurable, Byzantine-fault-tolerant VM that runs MPC protocols. A 'cluster' is the set of Arx nodes that jointly execute a computation via secret-sharing; each node is one processor and must stake collateral (slashable). Only an MXE can decrypt data marked with the 'Mxe' ownership type. This is the unit you target with a 'cluster offset'.
- [high] Terminology: Arcis, #[instruction], computation definitions, callbacks: Arcis = the Rust framework/DSL for confidential logic. Functions in an #[encrypted] module marked #[instruction] compile into individual circuits callable on-chain. Encrypted params use Enc<Owner,Data> where Owner is Shared (client + MXE can decrypt) or Mxe (only MXE). 'Computation definitions' are accounts initialized once per program (offset derived via comp_def_offset() from the fn name) that hold instruction metadata. Results return via a #[arcium_callback(encrypted_ix=...)] instruction the cluster invokes on your program AFTER the MPC finishes — i.e. the flow is asynchronous, not a synchronous CPI.
- [high] Integration with existing Anchor program: You keep an Anchor program but swap #[program] for the #[arcium_program] macro; per confidential computation you implement three Solana instructions (init comp-def, invoke, callback). This is directly relevant: FlipVault's existing Anchor program can be adapted rather than rewritten, with the FLIP math moved into Arcis circuits and the VRF/selection staying as ordinary public on-chain logic.
- [high] Toolchain & dev workflow: arcup installs the toolchain; arcium init scaffolds (creates encrypted-ixs/); arcium build compiles circuits + programs; arcium test runs TS tests; arcium localnet spins up a local Solana validator + Arcium nodes + a callback server for fully-local testing (auto-sets ARCIUM_CLUSTER_OFFSET); arcium deploy ships the MXE. ### Arcium developer model: authoring confidential computations (Arcis), the computation-definition + MXE account model, encrypted input/state representation, the queue/callback finalization pattern, and client encrypt/decrypt — applied to FlipVault path B. (devmodel)

CONFIRMED (current 2026): Arcium is a Solana confidential-compute network (MPC) that is live as Mainnet Alpha since ~Feb 2026 (permissioned cluster). The dev model is an extension of Anchor. You write MPC "circuits" in **Arcis** (a Rust DSL, crate `arcis` 0.9.6) inside an `encrypted-ixs/` crate, marking a module `#[encrypted]` and functions `#[instruction]`. Your normal Anchor program (deps `arcium-anchor`, `arcium-client`, `arcium-macros` all 0.9.6; `anchor-lang` 0.32.1) does NOT run the math — it QUEUES a computation to the Arcium program via `queue_computation(...)` CPI and gets the encrypted result back later via a CALLBACK instruction (`#[arcium_callback]` + `#[callback_accounts]`). The toolchain is the `arcium` CLI (managed by `arcup` 0.11.1; install via install.arcium.com), requiring Solana CLI 3.1.10 and Anchor 1.0.2; scaffold with `arcium init`, build with `arcium build`, test with `arcium test` (local docker MPC cluster). The TS client (`@arcium-hq/client`) encrypts inputs with x25519 ECDH + a Rescue cipher and decrypts its own results. CRITICAL FOR FLIPVAULT: persistent encrypted state IS supported — you store ciphertext as `[[u8;32]; N]` + a `u128` nonce in an ordinary Anchor account, feed it into the circuit as `Enc<Mxe, T>`, read-modify-write it, and the callback writes the new ciphertext back. Fixed-point `f64` (52 fractional bits) with `* / + -` is supported, so constant-product curve math is expressible. The whole FlipVault flip (buy/sell on curve, P&L, balance + curve-state update) maps cleanly to ONE stateful confidential instruction whose result is written back as ciphertext. Main caveats: async two-tx lifecycle (queue then callback, seconds of latency) is awkward for a 30s public VRF tick; 1232-byte output cap; no Vec/loops/dynamic indexing; mainnet is permissioned alpha; per-comp cost/latency numbers are not officially published.

**Findings:**
- [high] Arcis = the Rust DSL for authoring the confidential computation; lives in encrypted-ixs/ crate: You write circuits in a separate crate (default `encrypted-ixs`) depending on crate `arcis` (0.9.6). Pattern: `use arcis::*;` then `#[encrypted] mod circuits { use arcis::*; ... }`. Each confidential op is a `#[instruction] pub fn name(...) -> ...`. Confirmed verbatim from coinflip example: `#[instruction] pub fn flip(input_ctxt: Enc<Shared, UserChoice>) -> bool { let input = input_ctxt.to_arcis(); let toss = ArcisRNG::bool(); (input.choice == toss).reveal() }`. Inputs are decrypted into secret-shared form with `.to_arcis()`; encrypted outputs produced with `owner.from_arcis(value)`; public outputs with `.reveal()`. `ArcisRNG` provides in-MPC trustless randomness (bool/integers).
- [high] Two encryption ownership types: Enc<Shared,T> (client+MXE) and Enc<Mxe,T> (MXE-only) — the latter is how you persist private STATE: `Enc<Shared,T>` wraps {x25519 pubkey, nonce, ciphertext} and is for user-supplied data the user can also decrypt. `Enc<Mxe,T>` wraps {nonce, ciphertext} (no pubkey) — data only the MPC cluster can collectively decrypt — and is the recommended container for protocol state stored on-chain. New MXE-owned state is created in-circuit with `Mxe::get().from_arcis(value)`. This is exactly the FlipVault need: bonding-curve reserves/supply and per-box PERP/SOL balances live as `Enc<Mxe,...>` ciphertext.
- [high] Persistent encrypted state is stored in a NORMAL Anchor account as fixed byte arrays + nonce: Confirmed from the voting example: `pub vote_state: [[u8; 32]; 2], pub nonce: u128,` in `PollAccount`. Each confidential scalar is one `[u8;32]` field element; an N-field struct is `[[u8;32]; N]`. Generated callback output structs expose this as `MXEEncryptedStruct<LEN>{ nonce:u128, ciphertexts:[[u8;32];LEN] }` (or `SharedEncryptedStruct<LEN>{ encryption_key:[u8;32], nonce:u128, ciphertexts:[[u8;32];LEN] }`). The callback writes it back: `ctx.accounts.poll_acc.vote_state = o.ciphertexts; ctx.accounts.poll_acc.nonce = o.nonce;`. So FlipVault stores curve state + each box's balance as such arrays.
- [high] Computation Definition (CompDef) accounts hold the compiled MPC bytecode and must be initialized once per instruction: A CompDef account bridges your program to the MPC net. It has an Interface account (seeds `b"ComputationDefinitionAccount", mxe_program_id, comp_def_offset`) plus Bytecode accounts (seeds `b"ComputationDefinitionRaw", comp_def_acc, i`). `comp_def_offset = u32::from_le_bytes(sha256(instruction_name)[0..4])`; helper const e.g. `COMP_DEF_OFFSET_ADD_TOGETHER`, PDA via `derive_comp_def_pda!(OFFSET)`. Initialized on-chain by an instruction using `#[init_computation_definition_accounts("name", payer)]` that calls `init_computation_def(...)`. Large circuits use `CircuitSource::OffChain(...)` with `circuit_hash!` embedding the sha256 from `build/{circuit}.hash`.
- [high] MXE = the set of Arcium accounts your program references; several PDAs derived by macros: MXE (Multiparty-eXecution-Environment) accounts: `mxe_account` (`derive_mxe_pda!()`), `mempool_account` (`derive_mempool_pda!(mxe)`), `executing_pool`, `computation_account` (per-computation, keyed by a random `computation_offset`), `cluster_account` (`derive_cluster_pda!(mxe)`), plus an `address_lookup_table` (`derive_mxe_lut_pda!()`). The MXE program is YOUR Anchor program registered with the network. MPC cluster selection is configured via `Arcium.toml` (e.g. `[clusters.devnet] offset = 456`).
- [high] QUEUE pattern: your Anchor ix builds encrypted args and CPIs queue_computation, naming the callback: The queue instruction's Accounts struct is annotated `#[queue_computation_accounts("instruction_name", payer)]` and includes payer (Signer), a `sign_pda_account` (type `ArciumSignerAccount`), all the MXE/mempool/exec-pool/computation/cluster/compdef PDAs, and `arcium_program`. Body calls `queue_computation(ctx.accounts, computation_offset, args, vec![<Callback>::callback_ix(...)], <tx_count>, <priority_fee>)?`. Args built with `ArgBuilder`: for `Enc<Shared,T>` push `x25519_pubkey()`, then `plaintext_u128(nonce)`, then ciphertext via `encrypted_u8()/encrypted_u16()/...`; for `Enc<Mxe,T>` push only the nonce + ciphertext (ORDER MATTERS — wrong order fails silently).
- [high] CALLBACK pattern: MPC result returns asynchronously in a second tx via a verified callback: Define `#[arcium_callback(encrypted_ix = "name")] pub fn name_callback(ctx: Context<NameCallback>, output: SignedComputationOutputs<NameOutput>) -> Result<()>`. The `NameOutput` struct is auto-generated at `arcium build` time from the circuit's return type (PascalCase + `Output`, fields `field_0`, `field_1`, ...). You call `output.verify_output(&ctx.accounts.cluster_account, &ctx.accounts.computation_account)` (BLS signature check) to get the typed result, then write ciphertext back to your state account / emit an event. Callback Accounts struct uses `#[callback_accounts("name")]` and needs arcium_program, comp_def_account, mxe_account, computation_account, cluster_account, instructions_sysvar.
- [high] Client (TS) encrypt -> queue -> await -> decrypt flow: `@arcium-hq/client`: gen x25519 keypair (`x25519.utils.randomSecretKey()`/`getPublicKey()`); fetch cluster key via `getMXEPublicKey()`; `shared = x25519.getSharedSecret(priv, mxePub)`; `cipher = new RescueCipher(shared)`; `ct = cipher.encrypt([bigints], nonce)` with a random 16-byte nonce; call your Anchor ix passing ciphertext+pubkey+nonce+computation_offset; `await awaitComputationFinalization(...)`; decrypt own result with `cipher.decrypt([result], resultNonce)[0]` where the output nonce = input nonce + 1 (MXE increments by 1 after revealing inputs). - [high] Constant-product bonding-curve math IS expressible (fixed-point f64): Arcis types include u8..u128/i8..i128, bool, fixed-length arrays, structs, tuples, Option<T>, Pack<T> (bit-packing), and EMULATED `f64`/`f32` as fixed-point with 52 fractional bits, range [-2^75, 2^75) (out-of-range silently clamped). // ===== encrypted-ixs/src/lib.rs (Arcis circuit) =====
use arcis::*;
#[encrypted]
mod circuits {
    use arcis::*;

    pub struct BoxState { pub sol: u64, pub perp: u64, pub in_perp: bool }
    pub struct Curve    { pub reserve: u64, pub supply: u64 }
    pub struct FlipOut  { pub box_state: BoxState, pub curve: Curve }

    #[instruction]
    pub fn init_curve(seed: Enc<Mxe, Curve>) -> Enc<Mxe, Curve> {
        let c = seed.to_arcis();
        Mxe::get().from_arcis(c) // store as MXE-owned ciphertext
    }

    // ONE flip: force-buy if holding SOL, force-sell if holding PERP.
    // box_ct + curve_ct are read-modify-write encrypted STATE; fee_bps public.
    #[instruction]
    pub fn flip_box(
        box_ct: Enc<Mxe, BoxState>,
        curve_ct: Enc<Mxe, Curve>,
        fee_bps: u64, // public mechanism param (10% = 1000)
    ) -> Enc<Mxe, FlipOut> {
        let mut b = box_ct.to_arcis();
        let mut c = curve_ct.to_arcis();
        // fixed-point price = reserve/supply via f64 (52 frac bits)
        let r = c.reserve as f64; let s = c.supply as f64;
        if b.in_perp {
            // SELL: PERP -> SOL on constant product, realize P&L
            let dx = (b.perp as f64) * r / (s + b.perp as f64);
            let net = dx * (1.0 - fee_bps as f64 / 10000.0);
            b.sol = b.sol + net as u64; b.perp = 0; b.in_perp = false;
            c.reserve = (r - dx) as u64; c.supply = (s + b.perp as f64) as u64;
        } else {
            // BUY: SOL -> PERP
            let spend = b.sol as f64; let fee = spend * fee_bps as f64 / 10000.0;
            let dy = (spend - fee) * s / (r + (spend - fee));
            b.perp = dy as u64; b.sol = 0; b.in_perp = true;
            c.reserve = (r + spend - fee) as u64; c.supply = (s - dy) as u64;
        }
        Mxe::get().from_arcis(FlipOut { box_state: b, curve: c })
    }
}

### Arcium Performance + Cost (latency, throughput, fees) for a 30s confidential "flip" tick (latency)

Bottom line: a single confidential "flip" computation can very plausibly fit inside a 30s tick TODAY — the dominant costs are (a) Solana confirmation on ~2 transactions (queue/commission + callback), ~1–5s each, and (b) MPC compute itself, which for a small fixed-point bonding-curve flip is on the order of ~1s or less, plus a historical ~1s Diffie-Hellman shared-secret derivation that Arcium has now made cacheable. So a realistic end-to-end round trip for one flip is roughly ~3–12s, comfortably inside 30s with margin. THE REAL RISK IS NOT SINGLE-FLIP LATENCY — IT IS THROUGHPUT/RELIABILITY UNDER CONTINUOUS LOAD AND AT SCALE. Confirmed network throughput is only ~200,000 computations/day (~2.3 comp/s) as of mid-2026, and ~80% of that is a single app (ZINC). A naive design that submits one MPC computation PER PLAYER PER 30s tick (N players) does NOT scale: N=100 players => ~2 comp/s sustained = roughly the entire current network. Arcium's own dark-pool demo had nodes crash and orders back up indefinitely under heavy load on devnet — a direct warning. The architecturally safe design is ONE confidential computation per tick that ingests the public VRF selection and mutates the shared encrypted curve + the selected box's encrypted state (a single batched state transition), NOT one computation per box. Cost: fees are paid in SOL (not ARX), as a per-Computation-Unit base fee + dynamic priority fee; Arcium publishes NO concrete per-CU price as of mid-2026, so absolute SOL/flip cost is UNCONFIRMED and must be measured on testnet/mainnet-alpha. Many specific latency/throughput numbers are NOT published; the "<5ms" figure that appears is an internal MPC-network claim and is NOT the user-visible round-trip. Treat all absolute numbers as estimates pending your own benchmarking.

**Findings:**
- [high] Network throughput is the binding constraint, and it is currently small: Confirmed (Solana Compass, June 2026): Arcium crossed 1,000,000 lifetime confidential computations and runs 'more than 200,000 such computations per day' — i.e. ~2.3 computations/second network-wide — claimed to be '~20x the throughput of its nearest competitor.' Critically, 801,863 of the 1,000,000 lifetime computations came from ONE app (ZINC). So usable headroom for a new app submitting continuously is a fraction of ~2.3/s. A design that submits 1 MPC computation per player per 30s tick scales as N/30 comp/s: N=100 => ~3.3/s (exceeds current sustained network rate), N=1000 => ~33/s (far beyond). This, not single-flip latency, is the feasibility wall.
- [medium] Single-flip round-trip latency very likely fits in 30s: End-to-end = client encrypt (local, ms) + submit/commission tx (Solana confirm ~0.5–2s at 'confirmed', up to ~5–13s for finalized) + queue wait (variable, ~0 when uncongested) + MPC compute + callback tx (another Solana confirm). For a SMALL circuit (a constant-product buy/sell: a few fixed-point mul/div, a P&L diff, balance updates), MPC online time is on the order of sub-second to ~1–2s under Cerberus/Manticore when uncongested. Historical ~1s overhead per encrypted instruction came from re-deriving a Diffie-Hellman shared secret in MPC; Arcium shipped 'shared rescue keys' caching so repeat user<->MXE interactions reuse a cached key (a local lookup), reclaiming ~1s/instruction (they cite ~9s saved over 10 dark-pool orders). Net realistic estimate: ~3–12s per flip when the network is not congested. Inside a 30s tick with margin.
- [high] A flip requires (at least) TWO Solana transactions, so you pay Solana confirmation latency twice: Lifecycle (docs): define -> commission (an on-chain tx that queues the computation in the cluster mempool) -> MPC execution -> post-execution callback (a second on-chain tx where the cluster posts the result back, verified via threshold BLS signatures, which then invokes your MXE callback instruction). The TS SDK exposes awaitComputationFinalization(provider, computationOffset, programId, commitment). So Solana block time / confirmation dominates the floor: you cannot go faster than ~1 submit-confirm + MPC + ~1 callback-confirm. Budget ~2–10s of pure Solana confirmation overhead per flip independent of MPC speed.
- [high] Reliability under sustained load is unproven and has visibly failed before: Confirmed: Arcium's public dark-pool demo, under heavy traffic on devnet/testnet, had 'some nodes crash and get stuck processing orders indefinitely, causing a large backlog.' Arcium framed it as expected testnet behavior. For a game that submits computations EVERY 30s FOREVER, a stuck/crashed node or a growing mempool backlog directly threatens the tick cadence. You must design for: computation timeouts (the valid-before timestamp), failure callbacks (failure callbacks are 'always static' per docs), and graceful tick-skip/retry. Do NOT assume a flip always finalizes within one tick.
- [medium] Throughput scales by PARALLEL CLUSTERS, not by making one cluster faster: Arcium's scaling story is 'parallel execution across independent MXE/Prover clusters.' A single cluster has fixed throughput; the network grows by adding clusters. Implication for you: your app is pinned to a chosen cluster (cluster offset in config). Your per-tick throughput ceiling is that cluster's capacity, not the whole network's 2.3/s. This strongly favors a ONE-computation-per-tick batched design over per-player computations.
- [high] Fees are paid in SOL as base fee + priority fee, priced per Computation Unit (CU): Confirmed (docs/pricing-and-incentives + tokenomics): computation fees are paid in the chain's native token (SOL on Solana), NOT in ARX. Cost model = base fee (set to a 'minimally economically viable' level via epoch-based node-operator voting) + optional dynamic priority fee (to jump the queue under congestion). Execution cost is measured in CUs (a standardized unit of MPC work). Docs explicitly note base+priority are divided by CUs used, so 'combining computations would not affect the cost' per CU — i.e. batching many box-updates into one circuit does NOT save on raw compute cost, but DOES save on Solana per-transaction overhead and queue slots. You also pay normal Solana tx fees + priority fees on BOTH the commission and callback transactions.
- [high] No published absolute per-computation price exists as of mid-2026: Arcium's pricing docs describe the MECHANISM (per-CU base + priority, epoch voting) but publish NO concrete numbers — no SOL/CU, no lamports, no example flip cost. Therefore any absolute 'cost per flip' is UNKNOWN and must be measured empirically on mainnet-alpha. Reference point for scale economics: ZINC generated ~31,089 SOL protocol revenue on ~310,895 SOL volume off ~801,863 computations — but that is ZINC's 10% protocol fee, NOT Arcium compute cost, so it does not give you per-computation infra cost.
- [medium] The '<5ms latency' figure is internal MPC-network latency, not user round-trip: Some secondary sources cite '<5 milliseconds of latency' for Arcium. Context: that refers to inter-node MPC communication latency being low because nodes are not forced to be geographically distant — it is an internal protocol-round latency claim, NOT the queue->compute->on-chain-result round trip a player experiences. Do NOT model your 30s budget on 5ms. Use the seconds-scale end-to-end estimate instead.
- [medium] MPC protocol choice trades security for speed; pick per your threat model: Cerberus = dishonest-majority, 'one honest node' correctness via MAC'd shares, abort-on-malice — strongest guarantee, the 'main backend.' Manticore = honest-but-curious, uses a Trusted Dealer for offline preprocessing, faster, weaker guarantee (suited to trusted-operator clusters / ML workloads). Preprocessing being done in advance means the online flip phase runs faster. For a money game where curve state must not be biased, Cerberus is the safer default even at some latency cost; Manticore is an option if your cluster operators are sufficiently trusted/bonded and you need the speed headroom.
- [medium] Hardware reqs imply clusters are commodity-CPU and not GPU-bound: Node setup (docs): 32GB+ RAM, 12+ cores @2.8GHz+, 1 Gbit/s min bandwidth, minimal disk, GPU NOT required; QUIC transport window tunables (ASYNC_MPC_STREAM_WINDOW_MB / CONN_WINDOW_MB) for MPC throughput. This means latency is bandwidth/round-trip-bound, not compute-bound for small circuits — good for a tiny flip circuit, and consistent with the sub-second-to-low-seconds online-phase estimate.

### Combining public VRF selection with Arcium confidential computation (private randomness vs public VRF; verifiability split) for FlipVault path B (vrf)

CONFIRMED FEASIBLE in 2026 with Arcium mainnet-alpha (SDK v0.5.1). Two independent facts make path B work. (1) Arcium has NATIVE MPC randomness inside circuits via `ArcisRNG` (e.g. `ArcisRNG::bool()`, `gen_integer_in_range`, `shuffle`) — the official `coinflip` example generates the coin toss entirely inside MPC, and a community `arcis-wheel` demo markets it as "provably fair randomness powered by Arcium MPC." So randomness CAN be generated/consumed inside the confidential computation. (2) An Arcium computation is queued by a normal Solana/Anchor instruction (`queue_computation(...)` which CPIs the Arcium program) and returns asynchronously via an `#[arcium_callback]` handler. There is nothing preventing the SAME instruction that consumes an external VRF result (ORAO/Switchboard callback or a Solana program reading the fulfilled randomness account) from then queuing the Arcium FLIP computation. So a public VRF result CAN trigger/feed an Arcium computation — through ordinary Solana composition, not a special Arcium-VRF integration (none exists/none found). RECOMMENDED ARCHITECTURE for "mechanism public, data private": keep the SELECTION public + verifiable using ORAO/Switchboard VRF over public/pseudonymous box identities (the index of the selected box, the VRF proof, the box pubkey are all public and on-chain-verifiable); then pass the selected box id as a PUBLIC plaintext arg into the Arcium FLIP circuit, which operates on the encrypted shared curve state (`Enc<Mxe, CurveState>`) and the encrypted per-box balance/position (`Enc<Mxe, BoxState>`), computes buy/sell on the bonding curve, updates P&L and balances confidentially, and writes back re-encrypted `Enc<Mxe,...>` ciphertext on-chain. The FLIP outputs nothing in cleartext (or only minimal public deltas you choose to reveal). DO NOT use ArcisRNG for the selection itself if you want public verifiability of fairness — ArcisRNG randomness is hidden inside MPC and is only "tamper-evident" via BLS-signed outputs (integrity), NOT publicly auditable as fair the way a VRF proof is. Use VRF for selection (publicly verifiable), use the MPC purely for confidential math (no randomness needed in the FLIP itself). The deposit/withdraw custody-privacy hiding of amounts is the harder part and is mostly orthogonal to selection (flagged in risks/open questions).

### Arcium Encrypted State + Scale (Path-B FlipVault): on-chain encrypted state model, per-player balances + shared encrypted curve, client-side read-own-state via re-encryption, mutation-by-computation flow, and N-player per-tick scaling/cost limits (current as of June 2026) (scale)

Arcium's model fits FlipVault path-B's "mechanism public, data private" goal, but with hard constraints that shape the architecture. CONFIRMED facts: (1) Arcium does NOT itself hold or mutate state. Encrypted state lives in normal Solana/Anchor accounts owned by YOUR MXE program, stored as ciphertext byte arrays (the docs' concrete example stores an encrypted result as `[u8;32]` per field element, plus a 16-byte nonce and a 32-byte x25519 pubkey for `Shared` data). (2) State is mutated by a "computation": your program CPIs `queue_computation` passing the current ciphertext as an input parameter (e.g. `Enc<Mxe, CurveState>`); the MPC cluster decrypts to secret shares with `.to_arcis()`, runs the fixed circuit (buy/sell math, P&L, balance update), re-encrypts the new state with `owner.from_arcis(...)`, and the result returns to an on-chain CALLBACK instruction that writes the new ciphertext back into the account. This read-modify-write is exactly the pattern in Arcium's `voting` (encrypted tally accumulated across many calls) and `order book`/dark-pool examples — strong evidence the FlipVault shared curve is buildable. (3) Reading "only your own state": store per-player data as `Enc<Shared, T>`; a computation re-encrypts (seals) a player's view to THEIR x25519 key via `recipient.from_arcis(value)`, and the player decrypts client-side with `RescueCipher(sharedSecret).decrypt(ct, nonce)` using a key that never leaves their machine (the `medical-records` example does exactly this re-encryption-to-recipient pattern). MXE-only protocol state (`Enc<Mxe,T>`) is decryptable only by the cluster — ideal for the shared curve reserves. KEY CONSTRAINTS for the 30s-tick game: (a) Callback OUTPUT must fit one Solana tx (~1232 bytes) → `OutputTooLarge` beyond that, so you cannot re-emit all N players' balances per tick; you mutate ONE box per tick (which matches the game loop) plus the curve. (b) Arcis has NO `Vec`/`HashMap`/`String` and only fixed-bound loops — so a single global "array of N encrypted balances" passed into one computation does NOT scale (dynamic indexing is O(n), arrays are fixed-length, and the whole struct must round-trip through the ~1232-byte tx on output). The scalable design is ONE PDA per player holding that player's `Enc` balance/position, plus ONE shared-curve account; each tick's computation takes (selected player's account ciphertext, curve ciphertext) → returns (new player ciphertext, new curve ciphertext). That keeps every computation O(1) in players. (c) Comparisons/division are EXPENSIVE in MPC (bit decomposition; both branches of an if/else always execute and cost is additive) — constant-product buy/sell with a fee and P&L sign check is doable but should be written carefully (fixed-point, hoist comparisons, avoid general division where possible). UNCERTAIN/STALE: published END-TO-END per-computation wall-clock latency on mainnet alpha is not documented — sources say "near real-time", "<5ms inter-node", and "10,000x faster than FHE" but give no concrete seconds. A 30s tick is almost certainly enough headroom for one small computation, but you MUST benchmark on devnet/mainnet-alpha before committing. Concrete SOL cost per computation is also NOT published (priced in stake-weighted Computation Units + priority fee); at one paid MPC computation every 30s (~2880/day) this is a real recurring operating cost that must be modeled.

**Findings:**
- [high] Arcium never holds/mutates state; YOUR Solana program does, via callbacks: Docs state 'The Arcium Network does not mutate any state itself.' State mutations happen inside the MPC computation, and the updated ciphertext must be returned and written back on-chain by your program's callback. The MXE 'program' is just a normal Solana smart contract you deploy (Anchor), plus computation definitions. Implication for FlipVault: your existing Anchor program structure largely carries over; encrypted values replace plaintext balances/reserves, and each flip becomes queue_computation + callback instead of a direct math instruction.
- [high] Encrypted state stored as ciphertext byte arrays in Anchor accounts (32 bytes per field element + nonce + pubkey): The callback-accounts example declares `#[account] struct SecretAdditionResult { sum: [u8;32] }` and writes `ctx.accounts.add_result_account.sum = o.ciphertexts[0];`. All Arcis values are stored as 256-bit Curve25519 field elements (32 bytes EACH regardless of whether it's a u8 or u128 — bounds enforced at compile time, not storage). For `Enc<Shared,T>` you also persist a 16-byte nonce and the recipient's 32-byte x25519 pubkey. Sizing: a per-player box holding {sol_balance:u64, perp_balance:u64, cost_basis:u64} = 3 field elements = 96 bytes ciphertext + nonce(16) + pubkey(32) + Anchor discriminator(8) ≈ 152 bytes/player PDA. Use the `Pack<T>` wrapper to compress several small values into fewer field elements and cut both storage and per-computation cost.
- [high] Mutation flow: queue_computation (input = current ciphertext) -> MPC to_arcis/compute/from_arcis -> callback writes new ciphertext: Confirmed read-modify-write pattern (voting example): input `vote_stats_ctxt: Enc<Mxe, VoteStats>`; `let mut s = vote_stats_ctxt.to_arcis();` (decrypt to secret shares, nodes never see plaintext); mutate (`s.yes += 1`); return `vote_stats_ctxt.owner.from_arcis(s)` (re-encrypt). Order-book/dark-pool example does the same with `Enc<Mxe, OrderBook>`. For FlipVault the tick computation signature is conceptually: fn flip(box: Enc<Shared,Box>, curve: Enc<Mxe,Curve>) -> (Enc<Shared,Box>, Enc<Mxe,Curve>) where the circuit branches on whether box holds SOL (force-buy) or PERP (force-sell), applies constant-product price + 10% fee, updates both, and re-encrypts box to the player and curve to the MXE.
- [high] Player reads ONLY their own state via re-encryption (sealing) to their x25519 key + client-side RescueCipher decrypt: Use `Enc<Shared,T>` for per-player data. Inside a computation, `recipient.from_arcis(value)` re-encrypts (seals) a value to a SPECIFIC recipient's public key (medical-records example: patient-controlled sharing; loan example: `loan_officer.from_arcis(is_eligible)` so only that party decrypts). Client side: derive sharedSecret via x25519 ECDH with the MXE cluster pubkey, then `new RescueCipher(sharedSecret).decrypt(ciphertext, nonce)`. The client private key never leaves the machine, so other players (and the indexer/keeper) cannot read a box's balance/P&L. This directly delivers FlipVault's 'balances/positions/P&L private, selection public' requirement.
- [high] Output size cap ~1232 bytes per computation (single Solana tx) => mutate ONE box + curve per tick, not all N: Docs: callback output must fit one Solana tx, ~1232 bytes max, else `OutputTooLarge`. You cannot return all N players' encrypted balances in one computation. This is FINE for FlipVault because the game already mutates exactly one selected box per 30s tick. The per-tick computation outputs only (updated selected box ciphertext, updated curve ciphertext) — well under the limit. Deposits/withdraws are separate per-player computations.
- [high] No Vec/HashMap/String and only fixed-bound loops => do NOT model the whole game as one big encrypted array; use one PDA per player: Arcis unsupported: Vec, String, HashMap, while/loop/break/continue/early-return, .reveal() inside non-constant conditionals. Supported: if/else, match, fixed-size arrays, structs, tuples, Option, fixed-bound for. Dynamic array indexing is O(n) (checks all positions). Therefore a single computation over an array of N players is both unscalable (O(n) indexing, both branches always execute) and blocked by the output-size cap. SCALABLE design: per-player PDA (player's Enc balance) + one shared-curve account; tick computation is O(1) in N. N scales by adding accounts, not by growing one computation. Solana account/rent cost is the practical N-limit, not Arcium.
- [high] MPC cost model: add/sub/mul-by-const ~free, mul cheap, comparisons expensive, general division VERY expensive, if/else cost = sum of both branches: Constant-product curve math (x*y=k style) is multiplication-heavy (cheap-ish) but the new price involves DIVISION (very expensive) and the buy-vs-sell branch + P&L sign check are comparisons (expensive, bit decomposition). Mitigations from docs: keep branches balanced, 'compute once reuse', hoist comparisons out of branches, prefer fixed-point and avoid general division (e.g. precompute reciprocals/scaled integer math, or restructure so the divisor is a constant/public quantity where the public-selection design allows it). The flip math is feasible but each tick is a non-trivial circuit — size it and benchmark.
- [medium] MPC backends: Cerberus (dishonest-majority, 1-honest-node, MAC-authenticated, abort-on-cheat, slower, stronger security) vs Manticore (honest-but-curious + trusted dealer, faster): For a money game you want Cerberus-class guarantees (malicious security: a cheating node is detected and the computation aborts rather than silently corrupting balances). Manticore trades that for speed (aimed at AI/ML). Cluster sizes/exact node counts per computation are NOT documented publicly. The one-honest-node assumption means privacy holds as long as a single node in the cluster is honest.
- [high] Selection stays public/verifiable while flip math is private — clean separation: FlipVault's VRF selection picks among PUBLIC/pseudonymous box identities (PDAs). That selection can remain fully on-chain and verifiable exactly as today. Only the consequence (the buy/sell math, balance/P&L/curve update) moves into the Arcium computation. This matches Arcium's 'public mechanism, private data' framing and avoids putting randomness inside MPC (though ArcisRNG exists if you ever want private selection).
- [medium] Custody privacy (hide deposit/withdraw amounts) is supported but adds computations and a confidential-balance bridge: Hiding deposit/withdraw amounts means the SOL amount entering/leaving a box must be encrypted. Arcium's CSPL (Confidential SPL) standard provides encrypted balances + hidden transfer amounts and acts as a bridge between Solana's transparent layer and MXEs. Native SOL is transparent on Solana, so to hide deposit AMOUNTS you likely wrap into a confidential-balance representation (CSPL-style) and run deposit/withdraw as their own computations that update the player's Enc balance. This is the least-mature/most-uncertain part — verify CSPL availability and SOL support on mainnet-alpha.

// ===== Arcis confidential instruction (encrypted-ixs/src/lib.rs) =====
// One box + the shared curve mutated per 30s tick. O(1) in number of players.
use arcis_imports::*;

#[derive(ArcisType, Copy, Clone)]
pub struct Box { sol: u64, perp: u64, cost_basis: u64 } // a player's private state

#[derive(ArcisType, Copy, Clone)]
pub struct Curve { sol_reserve: u64, perp_reserve: u64 } // shared, MXE-only

#[instruction]
pub fn flip(
    box_ct:   Enc<Shared, Box>,   // sealed to THIS player's x25519 key
    curve_ct: Enc<Mxe, Curve>,    // protocol-only state
) -> (Enc<Shared, Box>, Enc<Mxe, Curve>) {
    let mut b = box_ct.to_arcis();      // -> secret shares (nodes never see plaintext)
    let mut c = curve_ct.to_arcis();
    let holds_sol = b.sol > 0;          // expensive comparison; compute once

    // constant-product price w/ 10% fee. Keep BOTH branches (cost is additive),
    // select by secret condition. Prefer scaled-integer math over general division.
    // force-BUY branch:
    let buy_perp_out  = mul_div(b.sol, c.perp_reserve, c.sol_reserve + b.sol); // approx
    // force-SELL branch:
    let sell_sol_out  = mul_div(b.perp, c.sol_reserve, c.perp_reserve + b.perp);

    if holds_sol {
        c.sol_reserve  += b.sol;  c.perp_reserve -= buy_perp_out;
        b.perp = buy_perp_out * 90 / 100; b.cost_basis = b.sol; b.sol = 0;
    } else {
        c.perp_reserve += b.perp; c.sol_reserve  -= sell_sol_out;
        b.sol  = sell_sol_out * 90 / 100; b.perp = 0; // P&L realized vs cost_basis
    }
    // re-encrypt: box sealed back to the player, curve back to MXE
    (box_ct.owner.from_arcis(b), curve_ct.owner.from_arcis(c))
}

// ===== Solana program (Anchor) =====
// per-player PDA holds the ciphertext; one curve PDA. Tick = keeper VRF picks a box
// (PUBLIC), then CPIs queue_computation with that box's ciphertext + curve ciphertext.
#[account]
pub struct EncBox  { pub ct: [[u8;32]; 3], pub nonce: u128, pub owner_x25519: [u8;32] }
#[account]
pub struct EncCurve{ pub ct: [[u8;32]; 2], pub nonce: u128 }

// queue (success path): queue_computation(ctx, comp_offset, args /*box ct, curve ct*/,
//                                          vec![flip_callback...], ...)
#[arcium_callback(encrypted_ix = "flip")]
pub fn flip_callback(ctx: Context<FlipCb>, output: SignedComputationOutputs<FlipOutput>) -> Result<()> {
    let o = output.verify_output(&ctx.accounts.cluster_account, &ctx.accounts.computation_account)?;
    ctx.accounts.enc_box.ct    = o.field_0.ciphertexts; // new sealed box
    ctx.accounts.enc_box.nonce = o.field_0.nonce;
    ctx.accounts.enc_curve.ct  = o.field_1.ciphertexts; // new curve
    Ok(())
}

// -> player sees ONLY their own balance/P&L. Output stays <1232 bytes per tick.

### Confidential custody on Solana for hiding deposit/withdraw SOL amounts (Token-2022 Confidential Transfer, shielded pools, composition with Arcium) — current 2026 feasibility for FlipVault path B (custody)

Two distinct primitives exist in 2026, and they hide DIFFERENT things. (1) Token-2022 Confidential Transfer/"Confidential Balances" hides the AMOUNT of transfers and resting balances inside the token program via twisted-ElGamal ciphertexts + ZK proofs — but it does NOT hide deposit/withdraw amounts (deposit public->confidential and withdraw confidential->public both move a PLAINTEXT u64), does NOT hide which account/owner is involved, and does NOT break linkage. So for custody privacy of SOL in/out of your vault, Confidential Transfer alone gives you essentially nothing: the on/off ramp amounts are public. (2) Shielded pools (Umbra on Arcium, Privacy Cash/Tornado-style, Hush) DO hide deposit/withdraw amount AND break the deposit<->withdraw linkage using note/commitment + Merkle-tree + ZK-membership designs — this is the only family of approaches that meaningfully hides SOL custody amounts.

CRITICAL CURRENT-STATE CAVEAT: Token-2022 Confidential Transfer was DISABLED on mainnet on 2025-06-19 (epoch 805) after a Fiat-Shamir transcript bug in the ZK ElGamal Proof program allowed forged proofs. After ~6 audits (Code4rena, Least Authority, ZkSecurity, Qedit, Trail of Bits) it is re-enabled on testnet/devnet and ships to mainnet via Agave 4.0, which as of June 2026 is a mainnet upgrade candidate targeting full mainnet rollout (originally May 2026, appears slightly slipped). Treat mainnet confidential transfers as "imminent but not confirmed live" — verify the Agave 4.0 mainnet feature-gate activation before depending on it.

For FlipVault path B specifically: Arcium's C-SPL (Confidential SPL) standard (launched/launching Q1 2026, with Encrypted Token Accounts and "confidential smart token accounts" where a PROGRAM can hold an encrypted balance) is the natural composition target — it lets the encrypted PERP/SOL game balances live in the same Arcium trust domain as your confidential flip math, avoiding a cryptographic seam between Token-2022 ElGamal ciphertexts and Arcium MPC secret-shares. The honest framing for the user: you can hide per-player game balances/positions/PnL/curve state inside Arcium; you can hide custody AMOUNTS only via a shielded-pool deposit/withdraw on top (Umbra-style or your own Arcium shielded pool); but the vault address, the fact that a deposit/withdraw happened, and timing inevitably leak, and the practical anonymity ceiling is bounded by your anonymity-set size, not by the crypto.

This is heavy for a 30s-tick game loop.
- [high] Optional global auditor key is a built-in escrow/compliance backdoor: A mint can set a global-auditor ElGamal pubkey; when set, every confidential transfer must additionally encrypt the amount under the auditor key so the auditor's secret-key holder can decrypt all amounts. Relevant for FlipVault: it is OPTIONAL (you can omit it for max privacy), but if you ever want regulatory comfort you can add it. It does not weaken privacy vs the public, only vs the auditor.
- [high] Hiding deposit/withdraw SOL AMOUNTS requires a shielded pool, not Confidential Transfer: Shielded-pool designs are the only family that hides on/off-ramp amounts and breaks linkage: Umbra (on Arcium) uses a shielded pool of mixed user tokens, sealing deposit details (amount + recipient Umbra address) into a cryptographic note/commitment; withdrawal proves ownership of one note via ZK without revealing which, severing the deposit<->withdraw link, AND encrypts amounts so balances within the pool stay private. Privacy Cash is Tornado-style (fixed/variable commitments in a Merkle tree, ZK withdrawal). Hush converts deposits to jitoSOL inside a shielded pool. These give amount + linkage privacy; Confidential Transfer gives neither at the boundary.
- [medium] Arcium C-SPL is the natural composition target for path B (single trust domain): Arcium announced C-SPL (Confidential SPL) at Breakpoint 2025, launching Q1 2026: encrypted balances + hidden transfer amounts for any Solana token, via Encrypted Token Accounts (ETAs, balances stored as ciphertexts) and — key for a game — 'confidential smart token accounts' where an on-chain PROGRAM can hold a confidential balance. - [medium] Token-2022 Confidential Transfer and Arcium do NOT natively share encrypted state: Inference grounded in the crypto: Token-2022 confidential balances are twisted-ElGamal ciphertexts verified by the on-chain ZK ElGamal program; Arcium operates on additive/Shamir secret-shares across MPC nodes under Cerberus. There is no documented primitive that lets the Arcium MPC directly compute on a Token-2022 ElGamal ciphertext or vice-versa. To compose them you would need an explicit bridge (decrypt-in-MPC then re-commit, or a ZK proof linking the two encodings), which adds latency and a new trust assumption. For a 30s loop this argues strongly for staying entirely inside Arcium (C-SPL) rather than using Token-2022 confidential balances for the game state.
- [high] The practical privacy ceiling: amounts/balances/PnL hideable; vault address, deposit/withdraw occurrence, and timing inevitably leak: What you CAN hide: per-player balances, positions, PnL, curve state (inside Arcium), and — with a shielded pool — deposit/withdraw amounts and the deposit<->withdraw link. What inevitably LEAKS: (1) the vault/pool/program address is public (it's where SOL lands); (2) the FACT that a deposit or withdrawal occurred at a given slot is public (a tx touched the pool); (3) timing/metadata — slot, fee-payer, compute, and tx graph; (4) anonymity is bounded by anonymity-SET size, not crypto — if only one player deposits in a window, the shielded amount/link is trivially inferable; (5) SOL must enter from a funded fee-payer, so the funding wallet's prior history is a deanonymization vector unless freshly funded. The honest ceiling: 'mechanism + custody addresses public, amounts/positions/PnL private, linkage private only within a sufficiently large anonymity set.'
- [medium] Native SOL needs wrapping; there is no native-SOL confidential balance: Token-2022 confidential transfer operates on SPL/Token-2022 mints, not lamports. SOL must be wrapped (wSOL native mint So111...112, syncNative) into a Token-2022 mint with the extension to be shielded — and the native wSOL mint is the legacy SPL Token program, so you'd need a Token-2022 wSOL-equivalent mint. // CONFIDENTIAL (Arcium encrypted-ixs / Arcis circuit over persistent MXE state):
//   - EncryptedCurveState { sol_reserve, perp_reserve }   // shared, encrypted

// #[encrypted] mod flipvault {
//   // called via queue_computation CPI after VRF picks a box
//   pub fn flip(curve: &mut EncCurve, box_: &mut EncBox, fee_bps: u64) {
//     if box_.holds_sol() {                 // force-BUY into PERP
//        let dx = box_.sol;                 // x*y=k constant product
//        let dy = curve.perp - curve.k() / (curve.sol + dx);
//        let fee = dy * fee_bps / 10_000;   // 10% -> treasury (encrypted)
//        curve.sol += dx; curve.perp -= dy;

//        box_.sol = dx - fee; box_.perp = 0; // pnl = sol - cost_basis (stays encrypted)
//     }
//     // treasury fee accumulates in an encrypted treasury balance (C-SPL)
//   }

**Risks:**
- [high] MPC cluster liveness gates the game loop: Arcium is permissioned in Alpha; a stalled/slow cluster stalls every 30s flip and could lock player funds mid-flip. -> Add keeper-side timeout + public 'flip pending/retry' state; never lock funds on a single un-acked computation; keep the existing fully-on-chain trustless version as a fallback mode; track Arcium decentralization (staking/slashing, TGE) before mainnet money.
- [medium] Throughput/latency: ~1s+ per-request overhead and ~2.3 comps/sec aggregate network capacity in mid-2026; folding deposits/withdrawals + flip math through MPC could blow the 30s budget or contend with other apps. -> Keep selection + accounting public; send ONLY the minimal flip math to MPC; exploit cached shared Rescue keys (per Arcium optimization); batch where possible; benchmark on devnet before committing cadence; consider >30s ticks if needed.
- [medium] C-SPL maturity: confidential token standard is Q1-2026-targeted and its dependency on the (currently disabled) Token-2022 ZK ElGamal program is unclear — private deposit/withdraw custody may not be production-ready when you build. -> Verify C-SPL mainnet status directly before depending on it; have an interim where balances are encrypted inside Arcium MXE state (not C-SPL) and custody amounts are only partially hidden; monitor ZK ElGamal re-enable.
- [medium] Wrong randomness surface: using Arcium's internal MPC RNG for selection would make the lottery confidential/unverifiable, contradicting 'mechanism public'. -> Use a dedicated public VRF (Switchboard/ORAO) for box selection on-chain in the clear; feed only selected_box_id into the confidential circuit.
- [medium] Information leakage via metadata/timing even with encrypted balances (which box flipped, gas, callback timing, treasury growth) can deanonymize positions/P&L. -> Keep treasury balance encrypted (C-SPL); avoid emitting amount-correlated events; consider decoy computations or fixed-size callbacks; threat-model the public selection + public timing channel explicitly.
- [medium] Trust-assumption immaturity: Cerberus (safe, dishonest-majority) is slower; teams may be tempted onto Manticore (honest-but-curious) for speed, weakening guarantees for a money game. -> Pin the circuit to Cerberus for any value-bearing math; document the chosen protocol; do not trade security for latency on financial state.
- [high] Vendor/network single-point dependency: Path B couples a live money game to one young, permissioned network with minimal production track record. -> Ship Path B as an OPT-IN privacy mode layered on the proven on-chain version; gate mainnet-with-real-funds on Arcium full decentralization + audits; keep an exit path back to the transparent version.
